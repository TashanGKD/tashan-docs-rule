# 第一章：为什么要分三层

## 1.1 核心问题

设计一个多智能体系统时，我们会产生大量要求："agent 的 energy 不能超过 100"、"通知必须有 TTL"、"LLM 不能直接输出完整 ActivityInstance"……

这些要求从何而来？哪些是**不可删减的本质要求**？哪些是**当前实现方案带来的附属要求**？哪些是**工程现实的补丁**？

如果分不清楚这三类，就会产生两种错误：
- **错误一：把工程补丁误当成本质要求**，导致理论变得臃肿，系统换一种实现方案时不知道哪些要求还成立。
- **错误二：把本质要求降格为可选项**，导致系统看似工作，实则在某些条件下必然失败。

三层体系正是为了解决这个问题。

## 1.2 三层的含义与判据

```
T 层  ──  本质目标层
          ↓
PA 层  ── 抽象公理层
          ↓
G 层  ──  当前系统目标层 + 具体要求
```

**判断一个陈述属于哪一层，看它回答什么问题：**

| 层级 | 回答的问题 | 是否与实现绑定 | 是否对所有多智能体系统成立 |
|---|---|---|---|
| **T 层** | 系统最终要保什么性质？ | 否 | 是 |
| **PA 层** | 为了保住 T，结构上至少需要什么条件？ | 否 | 是 |
| **G 层** | 在我们的粒子体系中，PA 怎么落地？ | 是 | 否（仅对本系统）|

**例子**：

- "系统必须能持续演化"——这是 T 层（本质目标，与任何实现无关）
- "任何可消耗资源必须有恢复路径"——这是 PA 层（结构性条件，不依赖 energy 这个概念）
- "energy ∈ [0, 100]，且存在 recovering 类活动使 Δenergy > 0"——这是 G 层（在粒子体系中的具体落地）

## 1.3 三层之间的关系：抽象 → 具体化 → 实现

```
T 层    是目标的抽象集合      ← "系统要保什么"
  ↓
PA 层   是 T 的结构化约束     ← "为了保住 T，结构上至少需要..."
  ↓
G 层    是 PA 在当前粒子      ← "在 world/agents/meta 这个架构里，PA 长什么样"
         体系中的具体化
```

精确地说：

- PA 层不是凭空造出来的——每条 PA 都可以从 T 推导（如果不满足 PA，则 T 在某种条件下必然失效）
- G 层也不是随意设计的——每个 G 目标都是某条 PA 在特定架构约束下的具体化
- 如果一个 G 要求找不到对应的 PA，说明它是架构引入的附属要求，不应进入理论主干（这是 Gₘ 判准的依据）

## 1.4 物理类比：守恒律 → 动力学约束 → 具体系统方程

在物理学中：

- **守恒律（能量守恒、动量守恒）** = T 层——最抽象的本质约束，对任何物理系统成立
- **李雅普诺夫稳定性条件、哈密顿量的存在性要求** = PA 层——为了保住守恒律，系统的演化算子必须满足某些结构性条件
- **具体力学方程（$F = ma$、麦克斯韦方程组）** = G 层——在特定坐标系和特定物理模型下的具体化

一个物理学家不会把"$F = ma$ 是一个本质定律"和"能量守恒是一个本质定律"混为一谈——前者是 G 层（牛顿力学的具体化），后者是 T 层（任何物理系统的本质约束）。

我们的三层体系做的是同一件事，只是对象是多智能体系统而不是物理粒子系统。

# 第二章：T 层 — 本质目标

## 2.1 T 层的定义原则

进入 T 层的陈述必须满足：

1. **普遍性**：对任何可以被称为"多智能体系统"的系统成立
2. **不可绕过性**：如果违反，系统不是"跑得不好"，而是"不再是我们讨论的那类系统"
3. **无关实现**：不依赖 world/agents/meta 这类具体的架构概念

T 层目前有 5 个目标：T1-T5。

## 2.2 T 层的内在逻辑：每个目标回答什么问题

```
T1  ─────────── "系统能不能被描述清楚？"
  ↓ 前提
T2  ─────────── "系统在任意时刻是否处于合法状态？"
  ↓ 前提
T3  ─────────── "系统能不能一直跑下去？"
  ↓ 前提
T4  ─────────── "系统跑的是不是有意义的内容？"
    （T4 依赖 T3，但二者解决不同问题：T3 保证活，T4 保证活得有意义）
T5  ─────────── "主体与环境之间有没有真实的相互作用？"
    （T5 依赖 T1，但与 T2/T3/T4 是不同维度）
```

**T 层的串联依赖**：

- **T1 是前提**：若 T1 不成立（系统无法被形式化），后续 T2-T5 都失去意义。
- **T2 依赖 T1**：只有系统被定义了，才能讨论它的状态是否合法。
- **T3 依赖 T1+T2**：若系统未被定义或状态非法，不可能讨论其持续演化。
- **T4 依赖 T3**：若系统在有限时间内死亡，谈"开放演化"没有意义。
- **T5 并行于 T3/T4，但依赖 T1**：有效耦合需要系统被定义（T1），但不严格依赖 T3/T4。

**T3 与 T4 的关键区别**（容易混淆）：

> T3 = 系统活着（持续演化，不死亡）  
> T4 = 系统活得有意义（演化有语义进展，不僵死/不空转）

一个系统可以满足 T3（一直在运行）但违反 T4（永远在几个无意义状态间循环）。T4 进一步要求"活"的方式不是空转。

## 2.3 每个 T 目标的深度分析

### T1：可定义性

**回答的问题**：系统能不能被描述？

**定义**：系统必须能被形式化描述，至少能表达：(1) 可区分的主体与环境；(2) 可描述的状态；(3) 可描述的演化；(4) 可描述的相互作用。

**为什么必要**：如果连系统有什么、它的状态是什么、它如何演化都无法表达，那么"运行"、"正确性"、"稳定性"这些概念都失去着落。T1 不成立时不会得到一个"运行得差"的系统，而是根本没有系统。

**违反后果**：无法定义 agent、无法定义 world、无法定义耦合。"系统"退化为一团未经形式化的描述性语言。

**跨域类比**：
- 物理：等价于相空间（phase space）的存在性——没有相空间，就没有力学
- 计算机：等价于类型系统和形式规范——没有类型定义，程序是无法分析的字节流
- 芯片：等价于指令集架构（ISA）的定义——没有 ISA，CPU 是没有语义的硅片

---

### T2：合法性

**回答的问题**：系统在任意时刻是否处于合法状态？

**定义**：系统在任意时刻都必须处于合法状态域中，包括：状态值域合法、写边界合法（写操作的影响范围受限）、时间单调、真源唯一、接口完备。

**为什么必要**：系统即使在运行，如果其状态自相矛盾（如同一状态分量被两个写者竞争写入），后续的所有演化都失去可信语义。T2 失效不是"跑得慢"，而是"跑出来的结果不可信"。

**三维写合法性**（T2 的核心）：
- 值域约束：写的**结果**在合法范围内（如 energy ∈ [0, 100]）
- 写权唯一：写的**主体**唯一（不能多个组件同时直接修改同一状态）
- 写边界：写的**影响范围**有界（一次写操作不应越权影响其他无关状态分量）

**跨域类比**：
- 物理：守恒律（能量守恒、动量守恒）——合法状态域就是满足守恒律的状态集合
- 计算机：内存安全 + 类型安全——非法内存访问 = T2 违反
- 芯片：寄存器合法值范围 + 总线仲裁（防止多驱动冲突，即多写者竞争）

---

### T3：持续演化

**回答的问题**：系统能不能一直跑下去？

**定义**：系统必须能在理想条件下持续无限时间演化，不因内部机制缺陷在有限时间内耗尽、沉默、堵塞或爆炸。

**为什么必要**：一个在有限时间内必然停机的系统，只是一个短程过程，不是长期自治系统。

**最小要求的四个方向**：
- 资源不会只耗不补（能量、注意力等）
- 累积不会只增不减（通知、记忆、效应）
- 调度不会中断（next_wakeup 链不断）
- 演化函数对任意合法状态都能给出下一步（无死状态）

**跨域类比**：
- 物理：非平衡定态（non-equilibrium steady state）——系统持续耗散但靠外部输入维持，不趋向热寂
- 计算机：liveness 属性（程序会继续推进）+ bounded resource usage（无内存泄漏、无队列爆炸）
- 芯片：时钟永不停止 + 电源轨永不崩溃 + 无死锁（所有总线事务最终完成）

---

### T4：开放演化

**回答的问题**：系统跑的是不是有意义的内容？

**定义**：系统不仅要能持续运行（T3），还要避免在非预期条件下收敛到吸收态或陷入无语义进展的病态循环。

**与 T3 的区别**：

| | T3 | T4 |
|---|---|---|
| 系统还在运行 | ✓ | ✓ |
| 演化有语义进展 | 不管 | ✓ |
| 典型反例 | 完全停机 | 无限空转/僵死循环 |

**判断"非预期"的判据**：由系统设计层定义的进展投影 P(s) 来判定。若在有限步内 P(s) 没有净变化，且这不是合法的静息期（如正常休眠），则视为非预期吸收态。

**跨域类比**：
- 物理：系统避免进入非目标吸引子（attractor）——如动力系统中的不期望极限环
- 计算机：系统满足"进展性"（progress property），不只是 liveness——进展性比 liveness 更强，要求做有意义的事
- 芯片：流水线向前推进（不只是有时钟信号，而且每个时钟周期确实在执行有意义的指令，而不是无限等待）

---

### T5：有效耦合

**回答的问题**：主体与环境之间有没有真实的相互作用？

**定义**：主体必须真实地受到环境影响，并真实地影响环境。系统不是若干自循环模块的并列。

**最小要求**：合法事件最终能影响相应主体；合法动作最终能改变相应世界。

**为什么必要**：若主体与环境之间没有真实耦合，则"多智能体—世界系统"只是多个孤立状态机加一个空壳环境。这样的系统即使"运行"，也无法称为多主体系统。

**与其他目标的关系**：T5 依赖 T1（需要系统被定义才能定义耦合），但不严格依赖 T3/T4（系统可能很快死亡，但在活着期间有真实耦合）。

**跨域类比**：
- 物理：粒子—场相互作用（Action-Reaction，牛顿第三定律）——主体影响场，场反作用于主体
- 计算机：具有 I/O 能力的进程（不只是纯计算，还与环境交互）
- 芯片：双向总线事务（读 + 写均有效）+ 中断到达 CPU（环境影响主体）

---

## 2.4 T 层依赖图

```
T1（可定义性）
  │
  ├──→ T2（合法性）——"定义好了，才能说状态是否合法"
  │      │
  │      └──→ T3（持续演化）——"状态合法，才能谈持续"
  │               │
  │               └──→ T4（开放演化）——"能持续，才能谈有无意义"
  │
  └──→ T5（有效耦合）——"定义好了，才能说耦合是否真实"
          （T5 与 T3/T4 互相独立，但需要 T1 作前提）
```

# 第三章：PA 层 — 抽象公理

## 3.1 PA 层的定义原则

进入 PA 层的陈述必须满足：

1. **从 T 可推导**：如果不满足这条 PA，则 T 层的某个目标在某种条件下必然失效
2. **抽象性**：不依赖具体的实现（不能包含 world/agents/meta、FieldProfile 等概念）
3. **结构性**：描述的是系统必须具备的**结构性条件**，而非目标本身

PA 层是 T 层与 G 层之间的桥梁：它比 T 更具体（有结构含义），但比 G 更抽象（与实现无关）。

## 3.2 PA 层的内在分组逻辑

9 条 PA 公理可以按功能分为 4 组：

```
定义组（服务 T1 为主）：
  PA1 ── 系统结构存在且可区分
  PA5 ── 状态所有权唯一

边界组（服务 T2 为主）：
  PA2 ── 所有状态量有界
  PA6 ── 所有效应有终局

生存组（服务 T3 为主）：
  PA3 ── 资源可恢复
  PA4 ── 调度持续再进入

进展组（服务 T4）：
  PA8 ── 语义进展可定义且可逃离

耦合组（服务 T5）：
  PA7 ── 信息可达（观察侧）
  PA9 ── 行动有效（行动侧）
```

**组内的互补关系**：
- PA2 + PA3 共同构成资源的长时稳定性：PA2 说"不能无限增长"，PA3 说"不能单调耗尽"。两者缺一，资源公理不成立。
- PA4 + PA8 共同构成调度的语义有效性：PA4 说"系统总能被再次触发"，PA8 说"被触发的演化要有语义进展"。
- PA7 + PA9 共同构成有效耦合的双向闭环：PA7 是观察侧（信息到达主体），PA9 是行动侧（行动影响环境）。

## 3.3 每条 PA 公理的深度分析

---

### PA1：存在性与主体—环境可区分公理

**从 T 推导**：T1 要求系统能被形式化描述，T5 要求有真实耦合。若系统内部不能区分"谁在行动"和"谁被作用"，则 T1 无法成立（无法形式化），T5 也无法成立（无方向性的耦合不是耦合）。→ 因此必须有 PA1。

**数学表达**：

```
PA1：存在性与主体—环境可区分

∃ 状态空间 S: S 良定义（非空、可数或连续、有描述）
∧ ∃ 分解: S = S_agent ⊕ S_world ⊕ S_interaction
  其中 S_agent ≠ ∅, S_world ≠ ∅, 两者可区分
∧ ∀ s ∈ S_legal, ∃ s' ∈ S_legal: transition(s) → s'  // 演化算子全定义
```

**含义解释**：
- `S_legal` = 系统的合法状态集合（不一定等于全状态空间）
- `S_agent ⊕ S_world` = 主体侧状态与环境侧状态可以被区分（⊕ 表示可分解）
- `transition(s) → s'` 全定义 = 对任意合法状态，都能给出下一个合法状态（没有死状态）

**为什么必要**：没有 PA1，就没有可供推导的基础。系统无法被形式化，后续所有公理都失去着落。

**违反后果**：无法定义 agent 和 world，无法定义演化，整个系统只是一堆无结构的状态转移。

**跨域类比**：
- 物理：相空间（phase space）的存在性 + 哈密顿算子的全定义性
- 计算机：进程模型（kernel space / user space 分离）+ 程序的语义完整性（任何合法输入都有输出）
- 芯片：ISA 定义（所有合法指令编码都有对应操作）+ 地址空间分区（内核/用户分离）

**具体违反案例**：某实现将 agent 的状态和 world 的状态混写在同一个 JSON 文件中，没有区分边界。两个 agent 的动作同时修改这个文件，产生不可预期的合并结果。无法定义"哪条记录属于哪个 agent 的本体状态"。这是 PA1 违反。

---

### PA2：有界性公理

**从 T 推导**：T2 要求系统任意时刻处于合法状态域（无界状态违反合法状态域）；T3 要求系统持续演化（无界增长最终导致资源枯竭或积压爆炸，系统在有限时间内死亡）。→ 因此必须有 PA2。

**数学表达**：

```
PA2：有界性

∀ X ∈ { R | R 在理论上可无界增长 ∧ 无界增长会破坏系统持续运行 }:
  ∃ B_X ∈ ℝ⁺:  ∀ t ≥ 0,  |X(t)| ≤ B_X

// X 包括：
//   资源类：energy, attention_budget, token_budget, ...
//   累积类：memory_size, notification_count, context_count,
//            obligation_count, queue_depth, ...
```

**含义解释**：
- "理论上可无界增长"：在系统设计阶段可识别出来，不要求运行时动态判定
- B_X 可以是有限上界（如 energy ≤ 100），也可以是有限表示（如通过压缩/遗忘机制保证 memory 不超过 context window）

**为什么必要**：任何无界增长的状态量，在足够长的时间后都会突破系统容量，导致运行停止。PA2 是 T3 的最小必要条件之一。

**违反后果**：资源枯竭、内部积压爆炸、被历史拖死、在有限时间内丧失演化能力。

**跨域类比**：
- 物理：有限体积系统中的能量有界（热力学极限下的有界性）
- 计算机：垃圾回收（GC）防止内存泄漏 + 队列深度限制（背压机制）
- 芯片：缓冲区大小限制（FIFO 深度上界）+ DRAM 容量上界

**具体违反案例**：某多智能体系统，每当一个 agent 发送消息，都会向所有其他 agent 发送通知，且通知不设 TTL。经过 1000 轮演化，50 个 agent 的系统中每个 agent 有约 50,000 条 pending 通知。assemble 上下文时无法在 context window 内容纳所有通知，系统无法正常演化。这是 PA2 违反导致 T3 失效。

---

### PA3：可恢复性公理

**从 T 推导**：T3 要求系统持续演化。若系统中的某类关键资源（如 energy）单调耗尽且没有恢复路径，则在资源耗尽后系统必然停机。PA2（有界性）只保证资源不超过上界，但不保证资源不低于下界。PA3 专门保证"从任意低状态都能恢复"。→ 因此 PA2 + PA3 合起来才保证 T3 的资源稳定性。

**数学表达**：

```
PA3：可恢复性

∀ R ∈ ConsumableResources（可被消耗的关键资源）:
  ∀ s 满足 R(s) = R_min（资源耗尽态）:
    ∃ path p: s →* s'  且  ΔR(p) > 0（沿路径 p，资源严格正增长）
    ∧ path p 在系统可达轨迹中存在（不只是理论存在，实际可达）
```

**含义解释**：
- `R_min`：资源的最低值（如 energy = 0）
- `ΔR(p) > 0`：严格正增长（不是"可以增长"，是"一定增长"）
- "在可达轨迹中存在"：恢复路径不能只在理论上存在，必须在实际的状态转移图中可达

**PA2 与 PA3 的互补关系**：

```
PA2: R ≤ B_R          ← 资源不能无限增长（上界）
PA3: ∃ recovery path  ← 资源不能单调耗尽（下界逃离）

两者合起来：资源在 [R_min, B_R] 内可持续波动
```

**跨域类比**：
- 物理：可逆热力学系统中的状态空间可达性——从任意低能态可以抽运到高能态
- 计算机：内存回收路径的存在（GC 不只有上界限制，还要能实际释放内存）
- 芯片：充电电路（charge pump）——即使电容放电到低电压，充电器可以恢复；电池的充电控制器

**具体违反案例**：系统设计 energy 有上界（满足 PA2），但恢复活动（recovering）只有在 foreground 场为 null 时才能触发，而 agent 长期处于 mission_locked 的会议场中无法退出。energy 一直在降，降到 0 后系统理论上应该切换到 recovering，但 mission_locked 约束阻止了这个切换。恢复路径"理论存在"但"实际不可达"，这是 PA3 违反。系统在有限时间内死亡（T3 失效）。

---

### PA4：持续再进入公理

**从 T 推导**：T3 要求系统持续演化；T5 要求有效耦合（环境变化影响主体）。若系统在某个状态下永远失去被再次触发演化的机会，则系统静默，T3 失效；且 agent 无法响应新的环境变化，T5 失效。→ 因此必须有 PA4。

**数学表达**：

```
PA4：持续再进入

∀ s ∈ S_reachable（任意可达状态）:
  ∃ t' > t_now: ∃ component C: scheduled_evolution(C, t')
  // 即：在任意可达状态下，系统中总存在某个将被触发演化的组件和时刻
  
// 注意：这不要求 s 必须变化（静息合法），
//        只要求"有机会演化"不被永久失去
```

**含义解释**：
- `scheduled_evolution(C, t')`：组件 C 在时刻 t' 有一次合法的演化机会（可来自时间驱动、事件驱动或内部调度）
- 这是一个**存在性条件**：只需存在一次未来的演化机会，不需要每时每刻都在演化

**PA4 与 PA8 的关键区别**：

```
PA4: 系统"有机会演化"——不等于"一定有进展"
PA8: 系统"演化产生语义进展"——是更强的要求

例子：
  PA4 满足的情形：agent 合法地休眠，next_wakeup = now + 8h，时间到了会被唤醒
  PA4 违反的情形：agent 的 next_wakeup = null，永远不会被唤醒
  PA8 违反的情形：agent 被不断唤醒，但每次都只是发"我在想..."的消息，没有实质进展
```

**跨域类比**：
- 物理：庞加莱递归定理（Poincaré recurrence）——有界系统最终会回到任意邻近状态附近
- 计算机：事件循环（event loop）永不停止 + 调度器总有下一个 tick
- 芯片：时钟信号（clock signal）永不停止 + 看门狗定时器（watchdog）防止系统"挂死"

**具体违反案例**：三个 agent A、B、C 约定：A 等 B 回复后才设置 next_wakeup；B 等 C 回复后才设置 next_wakeup；C 等 A 回复后才设置 next_wakeup。初始时，三个 agent 的 next_wakeup 都是 null（等待对方）。没有时间驱动的备用触发。系统永久沉默。这是 PA4 违反（三方循环等待，没有任何一个会被再次调度）。

---

### PA5：写权唯一性公理

**从 T 推导**：T1 要求系统可被形式化描述（若一个状态分量可被多方随意修改，它的值无法被单一地追溯，系统不可被形式化）；T2 要求状态合法（多写者竞争会导致状态污染和矛盾）。→ 因此必须有 PA5。

**数学表达**：

```
PA5：写权唯一性

∀ sᵢ ∈ S_durable（任意因果可追溯的状态分量）:
  |{ C | can_directly_write(C, sᵢ) }| = 1  // 直接写权限者唯一
  ∧
  ∀ C' ≠ authoritative_writer(sᵢ):
    influence(C', sᵢ) → must go through:
      C' → emit(event) → route → authoritative_writer(sᵢ).apply(event)

// "durable" 在 PA 层的抽象含义：
// 该状态分量的变化会影响系统的未来演化轨道（具有因果可追溯性）
// 不包括：临时计算缓存、传输中间值
```

**含义解释**：
- 写权限者唯一（**不是**"只能有一个读者"，而是"直接修改权限归一人"）
- 其他组件可以通过事件机制间接影响，但不能直接修改
- 这保证了每个状态变化都有唯一的责任主体

**跨域类比**：
- 物理：广义上的"唯一演化算子"——物理系统的时间演化由唯一的哈密顿量决定
- 计算机：Rust 的所有权系统（任意时刻最多一个可变引用）+ 数据库的 single-writer 原则
- 芯片：总线仲裁器（bus arbiter）——每个总线事务只有一个主设备（master）；内存保护单元（MPU）

**具体违反案例**：系统中，agent 的 active_key 既可以被 LLM 输出直接覆写，也可以被"ContextClosedEvent"处理逻辑自动重置。在并发场景中：LLM 刚输出 `active_key = "task_important"`，同一时刻 ContextClosedEvent 触发重置 `active_key = null`。两个写操作竞争，结果取决于顺序，且无法追溯哪一次写是"对的"。这是 PA5 违反（两个直接写权限者）。

---

### PA6：效应有终局公理

**从 T 推导**：T2 要求任意时刻状态合法（无终局的效应使系统状态不确定）；T3 要求持续演化（无终局的效应无限积压，最终耗尽资源）。→ 因此必须有 PA6。

**数学表达**：

```
PA6：效应有终局

∀ e ∈ Effects_generated（任意生成的效应对象）:
  ∃ t_terminal ∈ (t_created, +∞): status(e, t_terminal) ∈ TerminalStates(type(e))
  // 每个效应对象最终必须进入其类型对应的终态集合

// 各类型的终态集合：
// Notification:       { consumed, expired, merged, archived }
// Request/Attempt:    { accepted, rejected, failed, cancelled, applied }
// Task/Obligation:    { satisfied, cancelled, expired }
// ContextClose/Event: { applied, ignored, failed, archived }
```

**含义解释**：
- `t_terminal < +∞`：在有限时间内进入终态（不需要立刻，但不能永久悬挂）
- "终态"不是"消失"，而是达到一个明确定义的最终状态，之后不再产生新的系统行为

**跨域类比**：
- 物理：激发态粒子的自发辐射（不管多长时间，终将跃迁到基态）
- 计算机：RAII（Resource Acquisition Is Initialization）——资源生命周期有明确结束点；GC finalization
- 芯片：握手协议的完成（每个 bus transaction 的 ACK 最终到来或超时）

**具体违反案例**：某系统中，"帮助请求"义务（obligation）被创建后，提出者离开了该场景，切换到了另一个 context。原 obligation 没有超时机制，没有被显式取消，永远保持 "active" 状态。六个月后，系统中有 50,000 个永远活跃的 obligation。每次 agent 的 assemble 都要处理这些历史义务（义务注入是语义全量的），系统变得无法运行。这是 PA6 违反。

---

### PA7：信息可达性公理

**从 T 推导**：T5 要求有效耦合，"主体真实受到环境影响"。若合法应接收信息的主体永远接收不到该信息，则环境变化无法影响主体，T5 失效。→ 因此必须有 PA7。

**数学表达**：

```
PA7：信息可达性

∀ info i, ∀ agent aᵢ where authorized_recipient(aᵢ, i)（合法接收者）:
  ∃ t_delivery ≤ t_expiry(i):  received(aᵢ, i, t_delivery)
  // 信息在其有效期内到达合法接收者

// 注意：
// - 不要求"立刻到达"（可以有延迟）
// - 但必须在信息有效期 t_expiry 内到达
// - 对于无有效期限制的信息：∃ t_delivery < +∞ 即可（有限时间内总会到达）
```

**含义解释**：
- 这是一个**保证性条件**（liveness-style）：信息"最终会到达"
- 不是安全性条件（safety-style）：并非"不该到达的信息不会到达"

**跨域类比**：
- 物理：电磁信号在有限时间内传播到接收者（光速有限，但一定会到达）
- 计算机：可靠消息传递（reliable message delivery）——at-least-once + expiry；TCP 的数据到达保证
- 芯片：中断到达 CPU 的保证（中断信号不会无限期被屏蔽或丢失）

**具体违反案例**：一个紧急通知（semantic_class = hazard）被发往 agent B，告知 B 所在房间有烟雾警报。但系统的注意力门槛配置错误：hazard 类通知在 B 的 attention_policy 为 "direct only" 时被错误过滤（应强制通过）。B 永远收不到这条通知，继续在危险环境中"正常工作"。这是 PA7 违反（合法接收者永远未收到信息）。

---

### PA8：语义进展公理

**从 T 推导**：T4 要求系统不在非预期条件下陷入无语义进展的病态循环。若没有定义"什么是进展"、没有检测"是否在进展"的机制、没有逃离"无进展循环"的手段，T4 就无法成立。→ 因此必须有 PA8。

**数学表达**：

```
PA8：语义进展

// 前提：系统设计者定义一个目标相关的进展投影函数
∃ P: States → ProgressValue  // 进展投影

// 公理内容：
∀ cycle C ⊆ reachable_trajectories（可达轨迹中的任意循环）:
  if ∀ s ∈ C: P(s) = P_constant（循环内 P 不变，即无语义进展）
  then ∃ escape_mechanism(C):
    escape_mechanism 不依赖人工干预
    ∧ escape_mechanism 在有限时间内触发

// P(s) 是系统设计层给定的：
// - 强任务场：P(s) = (deliverables_submitted, agenda_updates, ...) 的变化
// - 弱社交场：P(s) = (commitment_changes, context_closures, ...) 的变化
// - P(s) 不由运行时主观判定，由设计者在部署时确定
```

**含义解释**：
- `P(s) = P_constant` 意味着在该循环中，进展函数没有变化——系统在"空转"
- `escape_mechanism` 必须自动触发，不能要求人工干预（否则在无人监督的自治系统中无效）
- P(s) 的设计是系统设计者的责任，不是运行时的主观判断

**PA4 与 PA8 的精确区别**：

```
PA4: ∀ s, ∃ t' > t_now: scheduled_evolution(s, t')
     = "系统不会永久停止被调度"
     
PA8: if ∃ cycle C where P(s) = const for all s in C:
       then ∃ escape_mechanism
     = "若系统在循环中无语义进展，则存在逃离手段"

PA4 保证：不永久静默
PA8 保证：不永久空转（虽然在动，但毫无意义的动）
```

**跨域类比**：
- 物理：动力系统中对病态吸引子的处理——系统不落入非目标极限环
- 计算机：分布式系统的进展性质（progress property）——不仅 liveness，还要有 meaningful progress；活锁检测与打破
- 芯片：流水线冒险检测（pipeline hazard detection）+ stall resolution——不只是"在工作"，还要"在有意义地推进指令"

**具体违反案例**：一个任务场中，5 个 agent 持续对话：A 说"我在思考"，B 说"好的继续"，C 说"我同意"……这个循环重复 200 次，`send_message` 被频繁执行，事件日志中充满了消息记录，系统看起来"很活跃"。但没有 deliverable 被提交，没有 agenda 被更新，没有 obligation 被满足。`last_progress_at`（进展时间戳）从未更新。在 PA8 缺失的系统中，这个循环可以永远持续，消耗无限计算资源和 token，而产出为零。这是 PA8 违反导致 T4 失效。

---

### PA9：观察—行动耦合公理

**从 T 推导**：T5 要求有效耦合，包含两个方向：主体受到环境影响（观察侧，由 PA7 保证）+ 主体的行动影响环境（行动侧，由 PA9 保证）。若 agent 只能观察不能影响，则耦合单向（不是真实耦合）；若 agent 只能行动不能观察，则行动是盲目的。→ 因此必须有 PA9 来保证双向闭环。

**数学表达**：

```
PA9：观察—行动耦合

∀ agent aᵢ ∈ Agents:
  // 观察侧（aᵢ 能观察到相关环境状态）
  ∃ ω_relevant ⊆ WorldState: aᵢ can_observe(ω_relevant)
  
  ∧
  
  // 行动侧（aᵢ 的行动能对世界产生净效果）
  ∃ α ∈ Actions(aᵢ):  
    execute(aᵢ, α) → net_effect(WorldState) ≠ ∅
    // "净效果"不为空：行动之后，WorldState 至少有一处实质性变化
```

**含义解释**：
- PA7 保证了"观察侧"（环境信息到达主体）
- PA9 保证了"行动侧"（主体行动真实影响环境）
- 二者合起来才构成双向真实耦合
- `net_effect ≠ ∅` 不要求"每个动作都成功"，但要求至少存在某个动作能产生实质世界效果

**PA7 与 PA9 的关系**：

```
PA7: info → agent（环境 → 主体，观察侧）
PA9: agent → world（主体 → 环境，行动侧）

T5 = PA7 ∧ PA9：双向真实耦合
```

**跨域类比**：
- 物理：作用力与反作用力（牛顿第三定律）——粒子既受到场的影响，也反作用于场
- 计算机：具有完整 I/O 能力的进程（不只是读，也能写）；双向通信接口
- 芯片：双向总线（bidirectional bus）——可以发出请求，也可以接受响应并产生结果

**具体违反案例**：一个"观察者 agent"被配置为只有 `read_messages` 权限，没有任何写操作权限。该 agent 可以看到所有消息，可以"认知"到群聊的进展，但无法发出任何有效行动（发消息、更新议程、满足义务等都被拒绝）。这个 agent 在系统中是一个纯旁观者——它自称是多主体系统的"成员"，但实际对世界状态没有任何影响。PA9 违反，T5 在该 agent 上失效。

# 第四章：G 层 — 当前系统目标

## 4.1 G 层的定义原则

进入 G 层的陈述必须满足：

1. **可追溯性**（Gₘ 判准）：必须能追溯到至少一条 PA 公理，进而追溯到至少一个 T 目标
2. **与当前架构绑定**：可以使用 world/agents/meta、FieldProfile、notification_buffer 等具体概念
3. **具体可验证**：要求应足够具体，使得实现者可以判断某个实现是否满足该要求

G 层是 PA 在"异步耦合粒子系统"这个特定架构中的具体化。

## 4.2 G 层的依赖链（核心内在逻辑）

**核心链（串行依赖）**：

```
Gₘ（裁剪元原则）
  ↓ "什么样的要求才能进入G层"
G0（存在性与本体结构）
  ↓ "系统被定义了"
G1（合法状态域 / 硬不变量）
  ↓ "系统在任意时刻都是合法的"
G2（单步演化完备性）
  ↓ "系统能完整地跑一轮"
G3（持续稳定性）
  ↓ "系统能永远跑"
G4（开放演化性）
  ↓ "系统跑的有语义意义"
```

**并行关系**：

```
G3 → G4（串行）
G0 + G2 → G5（语义有效耦合，与 G6 并行）
G0 + G1 → G6（一致性与可追溯性，与 G5 并行）
```

**特化关系**：

```
G3 + G6 → G7（认知连续性：G3+G6 在 LLM-based agent 上的特化）
```

**工程层**：

```
所有 G0-G7 → G8（工程鲁棒性：在现实计算机环境中实现 G0-G7 的增强）
```

**为什么是这个顺序？**

| 层次关系 | 理由 |
|---|---|
| G0 → G1 | 没有系统定义，无法定义"什么是合法状态" |
| G1 → G2 | 没有合法状态的边界，无法定义演化是否"完备" |
| G2 → G3 | 不能完成一步演化，不可能谈无限演化 |
| G3 → G4 | 系统不活着，谈"活得有意义"无从说起 |
| G5 ‖ G6 | 语义耦合和可追溯性解决不同维度的问题，逻辑上互不依赖 |
| G7 依赖 G3+G6 | 认知连续性需要稳定性（G3）保证记忆有界，需要可追溯性（G6）支持恢复 |
| G8 依赖全部 | 工程鲁棒性是对所有理想目标的现实增强 |

## 4.3 每个 G 目标回答什么问题（一句话定位）

| G 目标 | 回答的核心问题 | 具体化的 PA |
|---|---|---|
| **Gₘ** | 这条要求为什么能进 G 层？ | 判准（非 PA 具体化）|
| **G0** | 这个系统"是什么"？（本体结构）| PA1, PA5 |
| **G1** | 系统在任意时刻是合法的吗？（不变量）| PA2, PA5, PA6 |
| **G2** | 一次完整演化能跑完吗？（单步完备）| PA1, PA4, PA5 |

# 第五章：完整的 T→PA→G 推导链

## 5.1 从 T 到 PA：为什么这 9 条公理是必要的

每条 PA 都可以从 T 目标推导出来：

| PA 公理 | 服务的 T | 推导过程（为什么 T 需要这条 PA）|
|---|---|---|
| **PA1** | T1, T5 | T1 要求系统可被形式化 → 必须有可区分的主体/环境结构和全定义演化算子。T5 要求真实耦合 → 必须知道"谁在行动、谁被作用" |
| **PA2** | T2, T3 | T2 要求状态在合法域内 → 无界状态必然越界。T3 要求无限演化 → 无界增长在有限时间内导致停机 |
| **PA3** | T3 | T3 要求无限演化 → 资源不能单调耗尽（PA2 只保证上界，PA3 保证下界逃离）|
| **PA4** | T3, T5 | T3 要求无限演化 → 系统不能永久失去演化机会。T5 要求有效耦合 → agent 必须能响应环境变化（需要被再次触发）|
| **PA5** | T1, T2 | T1 要求可形式化 → 状态必须有唯一权威来源，才能被追溯。T2 要求合法性 → 多写者导致状态污染和矛盾 |
| **PA6** | T2, T3 | T2 要求合法性 → 永久悬挂的效应使状态不确定。T3 要求无限演化 → 无终局积压最终耗尽资源 |
| **PA7** | T5 | T5 要求有效耦合（观察侧）→ 信息必须能到达合法接收者，否则主体无法受到环境影响 |
| **PA8** | T4 | T4 要求开放演化 → 必须能定义、检测、逃离无语义进展的循环 |
| **PA9** | T5 | T5 要求有效耦合（行动侧）→ 主体的行动必须能产生世界效果，否则耦合单向 |

## 5.3 一条完整推导链的示例

**具体要求**：G3-R003（next_wakeup 永不为 null）

```
G3-R003 [next_wakeup 永不为 null]
  ↓ 属于哪个 G 目标
G3（持续稳定性）
  ↓ 具体化哪条 PA 公理
PA4（持续再进入）
  ↓ 服务哪个 T 目标
T3（持续演化）

推导链：
  T3 要求"系统能持续无限演化，不因内部机制而停止"
    → 持续演化的必要条件之一：系统必须在任意状态下都能在有限时间内重新获得演化机会
      → 这就是 PA4（持续再进入公理）
        → 在粒子体系中，time-driven 再进入的实现是：next_wakeup 时间戳
          → 若 next_wakeup = null，则 time-driven 链断裂，PA4 对应的再进入机会消失
            → 若 event-driven 也没有触发（如静默的群体场景），系统永久沉默
              → T3 失效
          → 因此必须：任意时刻，任意 agent 的 next_wakeup 必须是有效时间戳

违反后果（具体）：
  场景：一个任务群中，5 个 agent 完成了所有任务，last message 已经发出。
  如果没有 G3-R003：最后一个 LLM 输出没有指定 next_wakeup，系统没有默认填充机制。
  5 个 agent 的 next_wakeup = null，没有新的 event 到达。
  系统永久沉默。新的任务进来了，没有 agent 被调度处理。
  T3 失效（有限时间内系统停止演化）。
```

| **PA7** 信息可达性 | 电磁信号在有限速度内传播（信号终将到达，虽有延迟）| 可靠消息传递（TCP 的 at-least-once + 超时）| 中断到达 CPU 的保证（IRQ 不被无限期屏蔽）|
| **PA8** 语义进展 | 非线性动力系统中对病态极限环的避免（目标函数单调推进）| 分布式系统的活跃进展（progress property > liveness）+ 活锁检测与打破 | 流水线向前推进（not just busy, but making instruction progress）+ 死锁检测 |
| **PA9** 观察—行动耦合 | 牛顿第三定律（作用与反作用，粒子与场双向作用）| 完整 I/O 能力的进程（不只读，也能写）| 双向总线（read + write）+ 中断处理与寄存器写回的闭环 |

# 第七章：为什么这样分层，而不是别的分法

## 7.1 为什么需要 PA 层（不直接从 T 到 G）

如果直接从 T 目标推导 G 要求，会遇到以下问题：

**问题 1：同一个 G 要求服务多个 T 目标，无法清晰溯源**

例如 G1-R007（energy ∈ [0, 100]）：
- 它服务 T2（合法性，energy 超界是非法状态）
- 它也服务 T3（持续演化，energy 有界是资源稳定的前提）

如果直接从 T 跳到 G，这条要求就挂在两个 T 目标上，追溯路径变得模糊。但通过 PA2（有界性），可以清晰地说：G1-R007 具体化 PA2，PA2 同时服务 T2 和 T3。PA 层提供了清晰的中间汇聚点。

**问题 2：PA 层帮助发现 G 层的遗漏**

如果某条 PA 公理没有对应的 G 要求，说明该 PA 在当前架构中没有被正确实现——这是一个架构缺口。PA 层充当了完备性检查的"模板"。

**问题 3：PA 层允许不同的 G 层实现**

PA 层不绑定实现。如果未来换一套架构（不是 world/agents/meta，而是另一种实现方式），T 层和 PA 层仍然成立，只需要重新推导 G 层。G 层是可替换的，T 和 PA 是稳定的。

## 7.2 为什么 T 层恰好是这 5 个目标

这 5 个目标覆盖了"多主体系统正常运作"的所有必要维度，且互不重叠：

| 目标 | 覆盖的维度 | 若删除 |
|---|---|---|
| T1 | 存在与可描述性 | 系统连定义都没有 |
| T2 | 任意时刻的合法性 | 系统随时可能进入非法状态 |
| T3 | 时间维度的持续性 | 系统在有限时间内停机 |
| T4 | 演化的语义有效性 | 系统"活着"但不做有意义的事 |
| T5 | 主体与环境的耦合 | 系统是孤立状态机，不是多主体系统 |

如果删去任意一个，要么让某类根本性失败无法被防止，要么让体系无法完整描述多主体系统的本质需求。

**T4 的必要性**（容易被忽视）：

很多系统设计只关注 T1-T3 和 T5，忽略了 T4。但在 LLM-based 多智能体系统中，T4 的失效是极其常见的：

- agent 不断重复相似的输出（假活性）
- 义务循环依赖导致系统永久卡在一个"进展中"但实际无进展的状态
- group chat 中消息不断飞但没有任何 deliverable 产生

T4 正是为了防范这类"形式上活着，实质上已经死亡"的系统状态。

## 7.3 G 层为什么不是"PA 的直接翻译"

G 层是 PA 在特定架构（异步耦合粒子系统）中的**具体化**，不是字面翻译。具体化的过程包括：

1. **选择具体的状态量**：PA2 说"有界控制"，G 层选择 energy、memory、notification_buffer 作为具体的需要有界的量。这是架构决策。

2. **确定具体的实现路径**：PA4 说"持续再进入"，G 层确定具体通过 next_wakeup 时间戳 + 调度器来实现。这是实现决策。

3. **处理多对多映射**：同一个 G 要求可能具体化多条 PA（如 G2-R005a 具体化 PA5 + PA9）；同一条 PA 可能被多个 G 要求具体化（如 PA5 被 G0、G1、G6、G7、G8 的多个要求具体化）。

4. **识别系统特化目标**：G7（认知连续性）是对 LLM-based agent 的特化，不是所有多主体系统都需要的。它是 PA2+PA4+PA5 在特定主体类型上的专业化应用，不是新的 PA 公理的落地。

# 第八章：如何用这套体系审查一个新要求

当你遇到一个新的设计要求时，按以下步骤判断：

## 步骤 1：它回答什么问题？

```
"是什么"类问题 → 可能是 G0（存在性）
"任意时刻是否合法"类问题 → 可能是 G1（不变量）
"演化能不能完成"类问题 → 可能是 G2（单步完备）
"资源/累积是否有界"类问题 → 可能是 G3（持续稳定）
"有没有语义进展"类问题 → 可能是 G4（开放演化）
"耦合是否有效"类问题 → 可能是 G5（语义耦合）
"状态是否可追溯"类问题 → 可能是 G6（审计）
"LLM agent 认知连续性"类问题 → 可能是 G7（认知连续）
"工程实现是否鲁棒"类问题 → 可能是 G8（工程增强）
```

## 步骤 2：它具体化了哪条 PA？

对照 PA1-PA9 的定义，判断该要求是在保护哪个结构性条件。

如果找不到对应的 PA：
- 可能是该要求只与当前架构设计有关，不是本质必要的
- 可能是需要补充一条新的 PA 公理（比较罕见）

## 步骤 3：如果缺失，哪个 T 目标首先失效？

这帮助判断该要求的"严重程度"：
- 若 T1 失效 → 最严重（系统不可被定义）
- 若 T2 失效 → 严重（系统状态不可信）
- 若 T3 失效 → 严重（系统有限时间内死亡）
- 若 T4 失效 → 中等（系统活着但无语义意义）
- 若 T5 失效 → 中等（系统存在但无真实耦合）

## 步骤 4：是硬要求还是工程增强？

**硬要求**：在理想条件下仍然必须满足（对应 G0-G7 的要求）  
**工程增强**：只在非理想计算机环境下才需要（对应 G8 的要求，删掉后 G0-G7 的逻辑不坍塌）

如果一个要求被误归为"硬要求"但实际上是"工程增强"：理论文档变得臃肿，难以看清本质。  
如果一个要求被误归为"工程增强"但实际上是"硬要求"：系统在理想条件下也会失败，且问题难以被发现。

## 9.2 从具体要求到本质目标的完整追溯路径（示例）

**G8-R009（N-tick 超时逃生）**

```
G8-R009 [N-tick 超时逃生]
  → 属于 G8（工程鲁棒性）
    → 工程实现 G4-R003/004/005 的逃离机制
      → G4（开放演化性）
        → 具体化 PA8（语义进展公理）
          → 服务 T4（开放演化目标）

追溯链：
  T4 = "系统不能在非预期条件下陷入无语义进展的循环"
    → PA8 = "必须定义 P(s) 并存在逃离机制"
      → G4-R003 = "若 P(s) 无净变化循环存在，必须有非人工干预的逃离机制"（理想层要求）
        → G8-R009 = "N-tick 计数器 + 超时后注入 urgent 信号"（工程实现）

这说明：G8-R009 不是凭空设计的，它是 T4→PA8→G4→G8 整条逻辑链的工程实现端点。
```

**G7-R003（source_field_id 域隔离）**

```
G7-R003 [source_field_id 域隔离]
  → 属于 G7（认知连续性，LLM-based agent 特化目标）
    → 具体化 PA5（写权唯一性）在记忆写入上的应用
      → 服务 T3（持续演化）

追溯链：
  T3 = "系统持续演化"（对 LLM agent 来说，包括认知连续性）
    → PA5 = "每个 durable 状态分量有唯一写权限者"
      → G7（认知连续性特化）= "LLM agent 的 memory 必须有明确来源归属，防止跨场污染"
        → G7-R003 = "每条 MemoryEntry 必须携带 source_field_id，系统自动注入，LLM 不得覆写"

这说明：G7-R003 看似是一个具体的工程细节，但它背后有完整的逻辑：
PA5 → 记忆的写入必须有唯一的来源标注 → 这个来源标注防止 Memory Bleed → 这是 LLM agent
维持认知连续性（T3 在 LLM 系统中的特化需求）的必要条件。
```

# 第十章：总结

## 整个体系的一句话精髓

> **这套体系的核心思想是：把"为什么"和"是什么"分开。**  
> T 层回答"为什么"（系统最终要保什么），PA 层回答"结构上最少需要什么"，G 层回答"在这个具体架构里是什么"。  
> 任何具体要求都应该是这条链的末端——找得到它存在的理由，找得到它保护的本质目标。

## 三层体系的独特价值

| 价值 | 没有这套体系时的问题 | 有了这套体系后的解决 |
|---|---|---|
| **裁剪能力** | 不知道哪条要求可以删除 | Gₘ 判准：找不到 PA 追溯的要求，不进主干 |
| **完备性验证** | 不知道是否遗漏了什么 | PA→G 映射：每条 PA 都应在 G 层有具体化 |
| **层级分离** | 理想要求和工程补丁混写 | G0-G7 是理想层，G8 是工程层，明确区分 |
| **可移植性** | 换了架构，不知道哪些要求还成立 | T 层和 PA 层不与架构绑定，可以重新推导 G 层 |
| **风险评估** | 不知道缺失某条要求的后果 | 每条要求可追溯到 T 目标，知道违反的严重程度 |
| **解释能力** | 要求无法向外部解释清楚 | 任何要求都有完整的 T→PA→G 推导链 |
