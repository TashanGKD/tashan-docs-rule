# 《目标—公理—要求体系的验证映射表 v1》

> 版本：v1  
> 定位：本文件把“三层目标—公理—要求体系”转化为**可执行的验证映射**。  
> 目标：回答三个问题：  
> 1. 每一层条目究竟应由什么类型的任务去验证；  
> 2. 应依赖什么证据、什么判定逻辑；  
> 3. 若失败，应该首先怀疑哪类本质问题，并如何向上/向下追溯。

---

## 0. 使用说明

本映射表不是对原体系的重写，而是对原体系的**验证视角展开**。  
它必须同时满足以下三点：

1. **不改变原体系层级**：T 仍回答“最终目标”，PA 仍回答“抽象公理”，G 仍回答“当前体系要求”。
2. **不替代任务模板**：本文件只说明“测什么、靠什么测、失败后往哪里追”，不替代具体场景记录。
3. **支持跨体系扩展**：T/PA 的验证映射尽量通用；G 的验证映射只对当前体系成立。

---

## 1. 映射规则总览

## 1.1 基本映射对象

每个条目至少映射 6 类信息：

- **验证问题（Validation Question）**：这一条究竟要求证明什么。
- **主任务族（Primary Task Families）**：最适合用什么任务去验证。
- **核心证据（Core Evidence）**：什么证据最能支撑结论。
- **典型失败信号（Failure Signatures）**：失败通常长什么样。
- **向下落实接口（Downward Interface）**：往下应该落到哪些要求/机制/状态。
- **向上归因接口（Upward Interface）**：往上应该收束到哪些公理/目标。

## 1.2 映射的三个方向

- **自上而下**：T → PA → G → 任务 → 场景 → 证据
- **自下而上**：场景失败 → G 失效 → PA 失守 → T 受损
- **旁向审查**：当失败无法被现有链条解释时，转入“体系缺口审查”

## 1.3 证据使用原则

- **T 层**：不宜只靠单个局部场景判定，应更多依赖 PA/G 的聚合结论。
- **PA 层**：既看抽象逻辑，也看多个不同任务族上的重复表现。
- **G 层**：可由更直接的对象级、时序级、状态机级场景支撑。

## 2. 总体映射矩阵（总表）

| 层级 | 条目类型 | 核心问题 | 首选任务类型 | 首选证据 | 常见失败信号 | 失败后首查 |
|---|---|---|---|---|---|---|
| T | 本质目标 | 系统最终是否保住该目标 | GEN / HYB | 多任务聚合证据 | 多条 PA/G 同时失效 | 对应 PA 集 |

| G | 当前体系要求 | 当前实现是否落实该要求 | SYS / HYB | 对象级/时序级/状态级证据 | 明确违规、缺字段、断链 | 具体对象/规则 |

## 3. T 层验证映射表

## 3.1 T1 可定义性目标

### T1-VMAP

| 项目 | 内容 |
|---|---|
| 目标定义 | 系统必须可被形式化描述，能够区分主体、环境、状态、演化与相互作用 |
| 主验证问题 | 系统是否存在清晰的对象边界、状态空间、真源、演化路径与作用接口 |
| 主任务族 | GEN-LEGAL、GEN-TRACE、SYS-ONTOLOGY、SYS-STATESPACE |
| 核心证据 | 状态空间定义、真源表、粒子/对象结构、合法演化链、对象间边界 |
| 典型失败信号 | 主体/环境不可区分；真源不明；关键对象漂浮在体系外；旧新结构并存导致不可判定 |
| 向下落实接口 | PA1、PA5；当前体系中典型落点为 G0、G6 |
| 向上聚合逻辑 | 若 PA1 或 PA5 长期失效，则 T1 受损 |
| 备注 | T1 更适合通过“结构审计 + 代表性场景”共同判定，而非纯动态场景 |

### T1 推荐任务簇

- 主体—环境区分审计
- 状态空间分层审计（global / durable / runtime）
- 真源唯一性审计
- 演化四阶段闭环审计
- 跨粒子边界写入禁令审计

---

## 3.2 T2 合法性目标

| 项目 | 内容 |
|---|---|
| 目标定义 | 系统在任意时刻都必须处于合法状态域内 |
| 主验证问题 | 是否存在越界值、越权写、双真源、非法 effect 状态、时间倒流、接口旁路 |
| 主任务族 | GEN-LEGAL、SYS-INVARIANT、SYS-TRACE |
| 核心证据 | 不变量检查、状态快照、写回路径、真源表、effect 生命周期 |
| 典型失败信号 | baseline 被污染、energy 越界、直接跨粒子写、pending 永不终局、runtime 结构被误当 durable |
| 向下落实接口 | PA2、PA5、PA6；当前体系主要落点为 G1、G6 |
| 向上聚合逻辑 | 任一硬不变量持续失效，T2 至少降为 P2，严重时 P4 |
| 备注 | T2 是最适合用“单步推演 + 不变量检查”验证的目标层条目 |

### T2 推荐任务簇

- 写隔离任务
- 基线不变量任务
- 能量/资源有界任务
- effect 终态任务
- Event Log / Ledger 真源边界任务

---

## 3.3 T3 持续演化目标

| 项目 | 内容 |
|---|---|
| 目标定义 | 系统在理想条件下可长期持续演化，不因内部缺陷在有限时间内沉默、堵塞、爆炸或耗尽 |
| 主验证问题 | 是否存在再进入链、恢复路径、有界控制、终局控制、无死状态 |
| 主任务族 | GEN-BOUND、GEN-RECOVER、GEN-SCHEDULE、SYS-LONGRUN、SYS-ENGINEER |
| 核心证据 | next_wakeup 链、能量恢复轨迹、队列/记忆/义务有界性、长期场景轨迹 |
| 典型失败信号 | 无唤醒、无恢复、消息积压、上下文只增不减、重放后卡死、bounded 队列缺失 |
| 向下落实接口 | PA2、PA3、PA4、PA6；当前体系主要落点为 G2、G3、G8 |
| 向上聚合逻辑 | 若再进入、恢复、有界、终局四者中任一长期缺失，T3 至少降级 |
| 备注 | T3 必须看长时行为；单个短场景不足以给出完全结论 |

### T3 推荐任务簇

- 调度不断链任务
- 恢复路径任务
- 长时积压任务
- context 生命周期任务
- 工程故障恢复任务

---

## 3.4 T4 开放演化目标

| 项目 | 内容 |
|---|---|
| 目标定义 | 系统不仅要持续运行，还要避免非预期吸收态、语义空转、病态循环 |
| 主验证问题 | 是否定义了进展投影 P(s)；是否存在无净进展循环；是否有逃离机制 |
| 主任务族 | GEN-PROGRESS、GEN-LIVELOCK、SYS-PROGRESS、SYS-DEADLOCK |
| 核心证据 | progress projection、last_progress_at、状态循环轨迹、逃离触发机制 |
| 典型失败信号 | 形式上持续活跃但无语义推进；recovering/working 微震荡；永远 blocked 但不退出 |
| 向下落实接口 | PA8；当前体系主要落点为 G4，部分依赖 G3/G8 |
| 向上聚合逻辑 | 若无法区分“真推进”和“空转”，T4 不能判为完全满足 |
| 备注 | T4 的关键不是“系统有没有动”，而是“动是否形成目标相关净变化” |

### T4 推荐任务簇

- 空转循环任务
- progress projection 边界任务
- 微震荡与活锁任务
- 逃逸机制有效性任务

---

## 3.5 T5 有效耦合目标

| 项目 | 内容 |
|---|---|
| 目标定义 | 主体必须真实受环境影响，也必须真实改变环境 |
| 主验证问题 | 有权信息是否在有效期内可达；动作是否真正产生世界效果；两者是否形成闭环 |
| 主任务族 | GEN-COUPLE、GEN-DELIVERY、SYS-FIELD-ACTIVITY、SYS-RECEIPT、SYS-HAZARD |
| 核心证据 | local/external 路由、canInject/canWakeup、allowed actions、world-side apply、receipt |
| 典型失败信号 | 有权消息到不了主体；动作只停留在 agent 自述；低头族场景下物理场被忽略；hazard 无强制唤醒 |
| 向下落实接口 | PA7、PA9；当前体系主要落点为 G5，部分依赖 G2/G6/G8 |
| 向上聚合逻辑 | 输入侧和输出侧必须都成立；缺一不构成真实耦合 |
| 备注 | T5 极适合用“同一场景的双向闭环任务”验证 |

### T5 推荐任务簇

- 信息可达任务
- 动作世界生效任务
- 低头族与双场耦合任务
- hazard / evacuation 强制耦合任务

## 4. PA 层验证映射表

## 4.1 PA1 存在性与主体—环境可区分公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 状态空间良定义；至少能区分主体、环境与相互作用路径；演化算子对任意合法状态全定义 |
| 主验证问题 | 系统是否具备最小本体结构与演化定义；是否存在“合法状态但演化函数无定义” |
| 主任务族 | GEN-LEGAL、GEN-ONTOLOGY、SYS-STATESPACE、SYS-STEP-CLOSURE |
| 核心证据 | 粒子/对象定义、状态空间图、四阶段演化链、合法状态到下一步的映射 |
| 典型失败信号 | 缺少关键对象；合法状态下无法继续推演；状态空间层混淆 |
| 首查 G 接口 | G0、G2 |
| 失败根因提示 | 若对象、状态或演化任一缺失，多半是 PA1 失败而非纯局部 bug |

## 4.2 PA2 有界性公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 对任何可能无界增长的状态量，都必须有有限上界、有限表示或等价有界控制 |
| 主验证问题 | 系统是否识别并控制所有可无界增长的量 |
| 主任务族 | GEN-BOUND、GEN-LONGRUN、SYS-QUEUE、SYS-MEMORY、SYS-EFFECT |
| 核心证据 | 队列上限、TTL、分层记忆、compaction、bounded context count |
| 典型失败信号 | memory 无界、notification 无 TTL、context 只增不减、fanout 无上限 |
| 首查 G 接口 | G1、G3、G8 |
| 失败根因提示 | 概率性爆炸通常说明 PA2 没有真正落实，而不是“参数调小点就行” |

## 4.3 PA3 可恢复性公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 任何可消耗的关键资源，都必须存在从耗尽态可达的严格正恢复路径 |
| 主验证问题 | 是否能从资源耗尽态重新回到可演化态 |
| 主任务族 | GEN-RECOVER、SYS-ENERGY、SYS-INTERRUPT-RETURN、SYS-FAILOVER |
| 核心证据 | recovering 活动、sleeping/resting 正增益、恢复后的再进入轨迹 |
| 典型失败信号 | energy 归零后永久卡死；恢复态名义存在但不可达；恢复与工作微震荡 |
| 首查 G 接口 | G3、G5、G8 |
| 失败根因提示 | 若“有恢复路径但永远到不了”，也应视为 PA3 未满足 |

## 4.4 PA4 持续再进入公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 任意可达状态下，系统必须在有限时间内重新获得一次合法演化机会 |
| 主验证问题 | 是否总有 time-driven / event-driven / internal scheduling 之一保证再进入 |
| 主任务族 | GEN-SCHEDULE、GEN-RECOVER、SYS-WAKEUP、SYS-TIMER、SYS-INTERRUPT |
| 核心证据 | next_wakeup 非空、pending_wakeups、事件触发链、N-tick 逃生 |
| 典型失败信号 | 系统静默、next_wakeup 丢失、event 到了但无合法入口、等待链断裂 |
| 首查 G 接口 | G2、G3、G8 |
| 失败根因提示 | PA4 不要求本轮一定进展，但要求不能永久失去机会 |

## 4.5 PA5 写权唯一性公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 任意 durable 状态分量必须有且仅有一个直接写权限威者 |
| 主验证问题 | 是否存在双真源、旁路写、未定义共享写、View 持有新信息 |
| 主任务族 | GEN-LEGAL、GEN-TRACE、SYS-SOURCE-OF-TRUTH、SYS-LEDGER |
| 核心证据 | 真源表、写路径、apply 边界、Event Log 与 ledger 关系 |
| 典型失败信号 | 直接改别人 durable 状态；ledger 变第二真源；field/activity 旧新结构并存 |
| 首查 G 接口 | G0、G1、G6 |
| 失败根因提示 | PA5 是最常见的“表面功能正常、长期一定漂移”的根因之一 |

## 4.6 PA6 效应有终局公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 任何生成的 effect 都必须在有限时间内进入终态，不得永久悬挂 |
| 主验证问题 | 是否存在统一 effect 生命周期与终态；是否可观测终局 |
| 主任务族 | GEN-BOUND、GEN-LEGAL、SYS-NOTIFICATION、SYS-OBLIGATION、SYS-CLOSE |
| 核心证据 | TTL、终态集、归档/过期/合并机制、close path |
| 典型失败信号 | pending 永不结束、bounded context 无法关闭、request 无回执无失败终态 |
| 首查 G 接口 | G1、G3、G6、G8 |
| 失败根因提示 | 若 effect 没终局，不仅伤害合法性，也会拖垮持续演化 |

## 4.7 PA7 信息可达性公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 有权接收信息的主体，必须在有效期内收到该信息 |
| 主验证问题 | 路由、门槛、过滤、有效期与权限是否共同保证“该到的人能到” |
| 主任务族 | GEN-COUPLE、GEN-DELIVERY、SYS-LOCAL-EXTERNAL、SYS-HAZARD |
| 核心证据 | delivery chain、注入门槛、唤醒门槛、订阅/成员关系、TTL |
| 典型失败信号 | 权限合法但信息不到；background 门槛过高吞掉关键信号；hazard 无 override |
| 首查 G 接口 | G2、G5、G8 |
| 失败根因提示 | 输入不到并不一定是“耦合弱”，可能先是 PA7 失效 |

## 4.8 PA8 语义进展公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 必须定义目标相关的 progress projection；对无净进展循环必须提供逃离机制 |
| 主验证问题 | 是否能判定“推进/空转”；是否存在自动逃离活锁/吸收态机制 |
| 主任务族 | GEN-PROGRESS、GEN-LIVELOCK、SYS-BLOCKED-LOOP、SYS-TIMEOUT-BREAKER |
| 核心证据 | P(s) 定义、last_progress_at、timeout breaker、状态循环轨迹 |
| 典型失败信号 | 系统一直动但没有任务/义务/世界效果净变化；无限 blocked 循环 |
| 首查 G 接口 | G4、G8 |
| 失败根因提示 | 若只能依赖人工干预逃离，则 PA8 未满足 |

## 4.9 PA9 观察—行动耦合公理

| 项目 | 内容 |
|---|---|
| 公理定义 | 主体既能观察环境，又能通过行动对环境产生净效果 |
| 主验证问题 | 输入侧和输出侧是否同时成立，并形成闭环 |
| 主任务族 | GEN-COUPLE、SYS-FIELD-ACTIVITY、SYS-WORLD-EFFECT、SYS-RECEIPT |
| 核心证据 | observation path、allowed actions、world-side apply、receipt、可审计世界效果 |
| 典型失败信号 | 只能看不能改；只能发动作但世界无效果；动作成功只停留在自述 |
| 首查 G 接口 | G2、G5、G6 |
| 失败根因提示 | PA9 的验证必须至少包含一次“看见世界 + 改变世界”的闭环场景 |

## 5. G 层验证映射表（当前体系版本）

> 说明：G 层只对当前体系成立。  
> 当前依据：`G0 ~ G8` 的结构化要求层。  
> 本节不替代原 G 文档，只给出验证视角下的摘要映射。

## 5.1 G 层总览表

| G 条目 | 标题 | 直接服务对象 | 首选任务族 | 首选证据 | 典型失败信号 |
|---|---|---|---|---|---|
| G0 | 存在性与本体结构 | T1 / PA1 / PA5 | SYS-ONTOLOGY, SYS-STATESPACE | 真源表、目录/对象结构、状态分层 | 对象漂浮、层级混淆、旧新结构并存 |
| G1 | 合法状态域（硬不变量） | T2 / PA2 / PA5 / PA6 | SYS-INVARIANT, SYS-TRACE | 不变量检查、状态快照 | 越界、越权写、双真源、无终局 |
| G2 | 单步演化完备性 | T3 / PA1 / PA4 / PA5 | SYS-STEP-CLOSURE | assemble→evolve→emit→apply 轨迹 | 缺阶段、无 route、world effect 不落地 |
| G3 | 持续稳定性 | T3 / PA2 / PA3 / PA4 / PA6 | SYS-LONGRUN, SYS-BOUND, SYS-RECOVER | 长时轨迹、next_wakeup、恢复曲线 | 积压、断链、无恢复、context 不关闭 |
| G4 | 开放演化性 | T4 / PA8 | SYS-PROGRESS, SYS-LIVELOCK | progress projection、循环轨迹 | 空转、活锁、无逃逸 |
| G5 | 语义有效耦合 | T5 / PA7 / PA9 | SYS-FIELD-ACTIVITY, SYS-COUPLE | delivery、wakeups、effective actions、world receipts | 信息不到、动作不生效、低头族悖论 |
| G6 | 一致性与可追溯性 | T1/T2（审计侧） / PA5 / PA6 | SYS-TRACE, SYS-LEDGER, SYS-RECOVERY | Event Log、receipt、重放一致性 | 无可重放记录、ledger 冒充真源 |
| G7 | 认知连续性（系统特化） | T3（特化） / PA2 / PA4 / PA5 | SYS-MEMORY, SYS-RESUME | durable memory、resume token、跨轮连续性 | 重启失忆、恢复后断片、全量注入爆炸 |
| G8 | 工程鲁棒性 | 工程层支撑 G2~G6 | SYS-ENGINEER, SYS-FAULT, SYS-DISASTER | 重试、幂等、延迟、脑裂、灾害演化 | 掉线后写坏、重复投递、延迟破坏闭环 |

## 5.2 G0~G8 的验证提示

### G0 存在性与本体结构
- **首问**：对象是否“存在且归属清楚”？
- **必测**：
  - 粒子/对象最小骨架是否完整
  - `S_global / S_durable / S_runtime_total` 是否区分
  - 真源表是否闭合
  - 旧结构是否被明确禁止
- **首查失败**：多为 PA1 或 PA5 的结构缺陷

### G1 合法状态域
- **首问**：有没有“任一步后立刻非法”的情况？
- **必测**：
  - 写隔离
  - baseline 不变量
  - energy clamp
  - 单前景场与栈语义
  - effect 终局
  - Event Log 规范真源
- **首查失败**：多为 T2 直接受损

### G2 单步演化完备性
- **首问**：单步演化链条是否闭合？
- **必测**：
  - assemble 输入是否合法
  - evolve 是否产出合法提议
  - emit 是否只产事件不越权写
  - apply 是否写回自身并触发后续可观测效果
  - 是否存在 receipt 语义
- **首查失败**：多为 PA1/PA4/PA5/PA9 交叉问题

### G3 持续稳定性
- **首问**：系统能否一直跑，不被积压、断链、耗尽拖死？
- **必测**：
  - next_wakeup 断不断
  - energy 恢复是否可达
  - memory / notification / context 是否有界
  - bounded / ephemeral context 是否能关闭
- **首查失败**：多为 PA2/PA3/PA4/PA6 的任一未落实

### G4 开放演化性
- **首问**：系统是不是在“假活性”？
- **必测**：
  - 是否定义了 progress projection
  - 是否能区分“还在动”和“有语义推进”
  - 是否有 timeout breaker / escape rule
- **首查失败**：多为 PA8 直接失败

### G5 语义有效耦合
- **首问**：系统是不是一个真的“主体—环境耦合系统”？
- **必测**：
  - Field 与 Activity 是否分开建模
  - 行为是否由 `field × activity × agent_state` 共同决定
  - local / external 阈值是否分开
  - effective actions 是否由多重交集决定
  - 低头族与物理场动作是否被考虑
- **首查失败**：多为 PA7 或 PA9 失败

### G6 一致性与可追溯性
- **首问**：状态是否可重建、行为是否可审计？
- **必测**：
  - 所有 durable 变更是否进入 Event Log
  - receipt 与 writeback 是否可追
  - replay 是否一致
- **首查失败**：多为 PA5 或 PA6 审计侧失效

### G7 认知连续性
- **首问**：LLM 型 agent 是否拥有跨轮连续性？
- **必测**：
  - memory 是否 durable
  - 是否有 resume token / active key / current activity 连续性
  - 是否避免全量注入上下文
- **首查失败**：多为 PA2/PA4/PA5 在认知侧的系统特化失败

### G8 工程鲁棒性
- **首问**：非理想条件下系统是否仍可收敛，不写坏状态？
- **必测**：
  - 掉线/重试/重放/延迟/重复投递
  - 脑裂与 stale write
  - fanout / overload / bounded queues
  - 灾害类事件的强制路径
- **首查失败**：通常不是新增高层目标，而是对 G2~G6 的工程层支撑不足

## 7. 典型失败模式与上收路径

## 7.1 典型失败模式表

| 失败模式 | 表面现象 | 首查要求 | 上收公理 | 受损目标 |
|---|---|---|---|---|
| 双真源 | 两个位置都被当作权威状态 | G0/G1/G6 | PA5 | T1, T2 |
| 基线污染 | baseline_attention 被场/活动永久改写 | G1 | PA5 | T2, T3 |
| 挂起 effect | notification / obligation 永不终局 | G1/G3 | PA6 | T2, T3 |
| 唤醒链断裂 | next_wakeup 丢失或永远不触发 | G3 | PA4 | T3 |
| 恢复不可达 | 有 recovering 状态但永远到不了 | G3 | PA3 | T3 |
| 语义空转 | 反复演化但无净进展 | G4 | PA8 | T4 |
| 耦合失真 | 该到的信息不到，动作不生效 | G5 | PA7/PA9 | T5 |
| 动作无 receipt | agent 以为成功，但世界无确认 | G2/G6 | PA9/PA5 | T5, T2 |
| 掉线后写坏 | 重试/重放破坏 durable 状态 | G8/G6 | PA5/PA6 | T2, T3 |
| fanout 爆炸 | 大规模广播拖垮系统 | G3/G8 | PA2/PA6 | T3 |

## 7.2 上收路径模板

失败上收建议遵循：

```text
现象
  → 直接失败的 G 条目
    → 所映射的 PA 条目
      → 所服务的 T 目标
```

只有当这条链走不通时，才进入“体系缺口审查”。

## 8. 体系缺口的验证映射

本节回答：什么样的验证结果，不应只算“系统没做好”，而要进入“体系缺口审查”。

| 缺口类型 | 触发条件 | 最低证据要求 | 首选补法 |
|---|---|---|---|
| 无法归类型缺口 | 明确失败，但现有 T/PA/G 无法解释其为何算失败 | E2 | 补桥接公理或补适用域 |
| 双重失效无桥接型缺口 | 一个场景同时损害两个 T，但 PA 图上无连接路径 | E2 | 补中间 PA 或明确依赖关系 |
| 概率性失效无法收敛型缺口 | 多个任务均 P3，继续下钻后仍无法收束到更低层 | E2/E3 | 补稳定性类 PA 或细化现有 PA |
| 要求层补丁膨胀型缺口 | 只能不断加 G 层补丁，无法回收到 PA | E2 | 把共性上收为 PA |
| 互斥结论型缺口 | 不同任务对同一条目给出互斥而都合理的结论 | E2 | 明确适用域、前提或优先级规则 |

---

## 9. 验证执行顺序建议

推荐顺序如下：

### 第一阶段：T/PA 通用审计
先做：
- T1/T2 的结构与合法性审计
- PA1/PA5/PA6 的真源、边界与终局审计

> 原因：若真源、边界、终局不稳，后续动态结论都可能失真。

### 第二阶段：G 层闭环审计
再做：
- G2 单步闭环
- G3 持续稳定
- G5 耦合正确性
- G6 可追溯性

### 第三阶段：高阶动态审计
最后做：
- G4 开放演化
- G7 认知连续性
- G8 工程鲁棒性

### 第四阶段：体系缺口审查
只有在完成以上三阶段后，才对“原体系本身是否缺口”下结论。

## 10. 汇总输出建议

最终报告建议至少输出四张表：

1. **T 层汇总表**  
   逐项列出 T1~T5 的最终判定与证据等级。

2. **PA 层汇总表**  
   逐项列出 PA1~PA9 的最终判定、主要支撑任务、主要失败场景。

3. **G 层汇总表**  
   逐项列出 G0~G8 的落实状态、薄弱点、是否需修订当前体系。

4. **缺口审查表**  
   列出所有不能被现有体系稳定解释的情况及其建议修法。

## 11. v1 结语

本映射表的作用，是把“目标—公理—要求体系”从一套静态规范，变成一套**可以被系统性验证的判据网络**。  
它不替代原体系，也不替代任务与记录模板；它提供的是：

- 从高层条目到低层任务的桥；
- 从低层失败回到高层归因的桥；
- 从系统问题升级到体系缺口的桥。

因此，只要该映射表保持清晰，沙盘推演就不会退化成“想到什么测什么”，而会始终围绕一个可闭环、可追溯、可扩展的验证框架展开。
