# 异步耦合粒子系统：PA→G 正式映射表（v2.2 补完版）

> 基础文本：`三层目标—公理—要求体系_v2.2.md`
> 目的：把 v2.2 中已经给出的粗粒度 `PA→G` 映射，升级为一份可单独阅读、可校核、可追溯的正式映射文档。

---

## 0. 使用说明

这份文档只做一件事：

**把 PA 层 9 条抽象公理，完整映射到 G 层的具体系统目标与要求。**

这里的“完整”，不是把所有 requirement 逐条重抄一遍，而是把每条公理回答清楚：

1. 它在抽象上约束什么；
2. 它首先落到哪个 G 目标；
3. 它还通过哪些次级 G 目标得到补足；
4. 这些具体化分别在解决什么问题；
5. 如果缺失，会先坏在哪里。

因此，本文档的阅读顺序是：

`PA 的抽象语义 → G 的主具体化 → G 的次具体化 → 风险与校验`

## 2. 映射判定规则

为了避免“看起来像对应，实际上只是邻近”，这里把判定规则先钉死。

### 2.1 主具体化（primary concretization）

某个 G 目标若满足以下两条，则视为该 PA 的**主具体化**：

1. 它直接把这条 PA 的抽象语义落成系统级目标；
2. 去掉它后，这条 PA 在 G 层将失去主要承载者。

### 2.2 次具体化（secondary concretization）

某个 G 目标若满足以下至少一条，则视为该 PA 的**次具体化**：

1. 它不是这条 PA 的第一承载者，但对它做了补足；
2. 它把该 PA 从“原则成立”推进到“闭环成立”；
3. 它提供实现层、场景层或系统特化层的补充约束。

### 2.3 不计入 PA→G 主干的内容

以下内容不作为主映射：

- 仅属于文档书写规范的条目；
- 仅属于未来版本优化方向的条目；
- 只是在实现层提高性能，但不改变抽象成立与否的优化。

## 3. PA1 → G：存在性与主体—环境可区分

### 3.1 公理语义

PA1 回答的是：

> 这个系统到底能不能被形式化定义成“有主体、有环境、有相互作用路径”的系统？

其最小形式可以写成：

\[
\exists S_{agent}, S_{env}, I \quad \text{s.t.} \quad S = S_{agent} \times S_{env} \times I
\]

并且

\[
\forall s \in S_{legal},\; F(s)\;\text{is defined}
\]

也就是：

- 必须有主体侧状态；
- 必须有环境侧状态；
- 必须有两者之间的相互作用路径；
- 合法状态上演化函数不能没定义。

### 3.2 主具体化：G0

PA1 的主具体化是 **G0 存在性与本体结构**。

因为 G0 直接把“系统可被定义”落成了以下问题：

1. 系统由哪些本体实体构成；
2. 粒子最小形式是什么；
3. 系统级非粒子对象是否被承认；
4. 状态空间如何分层；
5. durable 真源分别放在哪里。

#### 3.2.1 关键承载条目

- G0-R001：世界粒子 A + 智能体粒子 B 的基本区分；
- G0-R002：粒子三元组 `P = (V_P, View_P, R_P)`；
- G0-R003：四子规则 `assemble → evolve → emit → apply`；
- G0-R004：系统级非粒子对象；
- G0-R005：`S_global / S_durable / S_runtime_total` 分层；
- G0-R014 ~ G0-R024：世界粒子与智能体粒子的最小骨架。

### 3.3 次具体化：G2 与 Gₘ

#### G2 的作用

G0 只是把“系统是什么”定义出来，G2 则补出：

> 这个被定义出来的系统，每一步到底怎么演化。

因此，PA1 不只落在“有对象”，还落在“演化函数在合法状态上有定义”。这部分由 G2-R001 / R003 / R006 承担。

#### Gₘ 的作用

Gₘ 不是系统性质，但它约束：

> 进入高层的概念必须真能参与系统形式化描述。

所以 Gm-R002、Gm-R003、Gm-R004 也在为 PA1 做边界防护。

### 3.4 若 PA1 失效，会先坏在哪里

第一类风险不是“运行慢”，而是：

- 你不知道谁是真正主体；
- 你不知道谁是环境；
- 你不知道某条状态属于谁；
- 你不知道一步演化到底以什么为输入、以什么为输出。

### 3.5 典型案例

**坏案例**：把 `FieldProfile` 当作独立顶层本体，同时又把它塞进 `interaction_contexts[*].field`，而且两边都能改。

后果：

- 本体归属不清；
- 真源位置不清；
- “场”到底是世界的一部分，还是独立本体，变得模糊；
- 后续 PA5 也会一起被拖坏。

## 4. PA2 → G：有界性

### 4.1 公理语义

PA2 回答的是：

> 哪些状态量可能无限增长，从而把系统拖死？

可写成：

\[
\forall X \in \mathcal{X}_{unbounded},\; \exists B_X < \infty \; \text{or bounded control } C_X
\]

关键点不在“都写死一个上界数值”，而在：

- 设计阶段必须识别潜在无界量；
- 对每类无界量都必须给出上界、有限表示或等价控制机制。

### 4.2 主具体化：G1 与 G3

PA2 的落地分成两半：

#### G1：任意时刻的状态边界

G1 承担“**每一时刻不能越界**”这部分。

代表条目：

- G1-R007：`energy ∈ [0,100]`；
- G1-R010 / R010a：effect 不得无限悬挂；
- 与合法状态域相关的所有硬不变量。

#### G3：长期演化中的累积有界

G3 承担“**长时间跑下去也不能爆**”这部分。

代表条目：

- G3-R006：memory 有界；
- G3-R007：禁止全量注入；
- G3-R008：notification_buffer 有界；
- G3-R011：bounded / ephemeral context 可关闭；
- G3-R012：最小无限演化必要条件集。

### 4.3 次具体化：G5 / G7 / G8

- **G5**：在义务、通知、可见切片上引入有界投影；
- **G7**：认知连续性并不意味着记忆可以无界，仍必须分层与压缩；
- **G8**：在真实工程里用 Sweeper、MAX_DELTA_T、overload、queue 等机制实现有界控制。

### 4.4 若 PA2 失效，会先坏在哪里

最先坏的是 T3，但也会反过来破坏 T2：

- memory 越来越大，assemble 上下文溢出；
- notification 永远堆积，系统再也清不掉历史；
- context 永不关闭，世界被旧交互对象拖死；
- agent 因全量注入而“看见一切”，最后什么也处理不了。

### 4.5 典型案例

**坏案例**：所有历史消息、所有 Obligation、所有记忆，都在每轮 assemble 里全量注入。

后果：

- 一开始似乎“信息更全”；
- 长时之后上下文爆炸；
- 触发成本越来越高；
- 系统形式上还活着，实质上已被历史拖死。

## 5. PA3 → G：可恢复性

### 5.1 公理语义

PA3 回答的是：

> 关键资源如果会消耗，那耗尽之后有没有路回来？

形式化骨架：

\[
\forall r \in \mathcal{R}_{critical},\; \forall s \in S_{depleted}(r),\; \exists \pi: s \rightsquigarrow s'\;\text{s.t.}\; \Delta r(\pi) > 0
\]

也就是：

- 不仅要存在恢复路径；
- 而且该路径必须从耗尽态可达；
- 并且恢复速度必须严格为正，而不是“名义上能恢复、实际上永远回不来”。

### 5.2 主具体化：G3

PA3 的主具体化是 **G3 持续稳定性**，尤其是：

- G3-R004：至少存在 recovering 活动预设，且从能量耗尽态可达；
- G3-R005：能量动力学写成场 × 活动 × 时间模型；
- G3-R012：把“存在严格正恢复路径”列入最小无限演化必要条件。

### 5.3 次具体化：G5 与 G8

- **G5**：恢复不是抽象恢复，而是经由具体活动与场组合落地；
- **G8**：Hysteresis Lock、sleep_debt 等防止系统在恢复阈值附近震荡或假恢复。

### 5.4 若 PA3 失效，会先坏在哪里

系统不会立刻坏，但会**不可逆地走向坏**：

- 每次行动都在消耗；
- 休息只是在“少消耗”，却没有“真恢复”；
- 长时间后所有 agent 都卡在低能量带；
- 调度链虽然存在，但再也无有效产出。

### 5.5 典型案例

**坏案例**：sleeping/resting 只是不产生新动作，但 energy 并不回升。

后果：

- 看起来系统支持“睡觉”；
- 实际上睡觉只是静止，不是恢复；
- 所有 agent 最终都掉到 0 能量，然后全体僵死。

## 6. PA4 → G：持续再进入

### 6.1 公理语义

PA4 回答的是：

> 系统在有限时间内，能不能重新获得一次演化机会？

形式化骨架：

\[
\forall s \in S_{reachable},\; \exists t' > t_{now}: Trigger(s,t')\;\text{is valid}
\]

注意：

- PA4 只要求“再被触发”；
- 不要求这一轮一定有语义进展；
- 它和 PA8 的区别就在这里。

### 6.2 主具体化：G3

G3 是 PA4 的主承载者：

- G3-R001：必须存在 time-driven 调度；
- G3-R002：必须存在 event-driven 调度；
- G3-R003：`next_wakeup` 永不为 null；
- G3-R012：把“调度链不断”列入最小必要条件。

### 6.3 次具体化：G2 / G4 / G5 / G7 / G8

- **G2**：单步四阶段是“再进入之后”如何完成一次演化；
- **G4**：软逃离信号需要再进入才能真正触发逃离；
- **G5**：`canWakeup`、用户直接输入、高危事件强制唤醒都在具体化“演化机会”的来源；
- **G7**：被打断之后还能重新回到原任务，是再进入在认知层的特化；
- **G8**：忙等防护、不应期、超时恢复，是 PA4 的工程抗失控实现。

### 6.4 若 PA4 失效，会先坏在哪里

最直接的故障不是“逻辑错”，而是：

- 系统卡住后再也不会动；
- 某 agent 静息后永远不被唤醒；
- 某任务没有外部事件时，系统永久沉默。

### 6.5 典型案例

**坏案例**：`next_wakeup = null`，且 agent 也没有待处理通知。

后果：

- 当前状态也许合法；
- 但未来再也没有触发器；
- 系统从“动态系统”退化成“静态快照”。

## 7. PA5 → G：写权唯一性

### 7.1 公理语义

PA5 回答的是：

> durable 真状态到底由谁写？

形式化骨架：

\[
\forall s_i \in S_{durable},\; |AuthWriter(s_i)| = 1
\]

并且

\[
\forall x \notin AuthWriter(s_i),\; x \to s_i \text{ must pass through the authoritative apply protocol}
\]

### 7.2 主具体化：G1 与 G0

#### G1：状态边界上的写隔离

G1 把“单一写权限威者”落成了硬不变量：

- G1-R001：跨粒子影响不能直接写 V_P；
- G1-R002：本体写权限边界；
- G1-R003：View 不持有新信息；
- G1-R012：唯一真源不变量。

#### G0：归属拓扑与真源位置钉死

G0 进一步回答：

- 这些真状态到底放在哪；
- `world / agents / meta` 如何分层；
- 哪些概念的 durable 真源路径已经钉死。

### 7.3 次具体化：G2 / G5 / G6 / G7 / G8

- **G2**：emit / apply 路径规定了“别人如何合法影响我”；
- **G5**：如 LLM 只能输出 activity preset，不能直接输出完整可写 JSON；
- **G6**：把写权唯一推进到“可审计唯一真源”；
- **G7**：记忆条目的归属和 source_field_id 自动注入；
- **G8**：split-brain 防护、拓扑排序、恢复 replay，都是写权一致性的工程补足。

### 7.4 若 PA5 失效，会先坏在哪里

这类错误最危险，因为系统表面上可能还能跑，但因果已经坏了：

- A 可以偷偷改 B 的 durable 状态；
- Event Log 记的是一套，物化状态是另一套；
- View 成了第二真源；
- LLM 自述“成功了”，世界却没有对应写入。

### 7.5 典型案例

**坏案例**：世界粒子直接把某 agent 的 `baseline_attention_policy` 改成更高阈值。

后果：

- 该 agent 的人格基线被环境永久污染；
- 以后退出当前场后，注意力阈值也恢复不了；
- T2 的合法性与 T3 的持续演化同时受损。

## 8. PA6 → G：效应有终局

### 8.1 公理语义

PA6 回答的是：

> 所有 effect 会不会永远挂着不结束？

形式化骨架：

\[
\forall e \in Effects,\; \exists \tau_e < \infty:\; state_e(t + \tau_e) \in Terminal(e)
\]

### 8.2 主具体化：G1 与 G3

#### G1：终态必须被明确定义

- G1-R010：Notification 必须有终局；
- G1-R010a：统一 effect 终态表必须存在；
- G1-R011：surfaced 与 acknowledged 不能混同。

#### G3：终局不仅要定义，还要能清掉历史

- G3-R008：notification_buffer 有界；
- G3-R009：最小终局实现；
- G3-R010：TTL 与 ack_required 后继迁移；
- G3-R011：bounded / ephemeral context 可关闭。

### 8.3 次具体化：G4 / G5 / G6 / G8

- **G4**：义务循环检测是在防止某类 effect 永久卡住；
- **G5**：`closeCondition`、ContextClosedEvent、生命周期语义属于 effect 终局在交互层的体现；
- **G6**：receipt、staging、writeback 顺序保证 effect 生命周期可审计；
- **G8**：Sweeper、恢复逻辑、工程清理机制保证终局真正落地。

### 8.4 若 PA6 失效，会先坏在哪里

- 通知永远 pending；
- 请求发出后没有接受/拒绝/失败；
- task 一直 active 但没有 satisfied/cancelled/expired；
- context 长年不关闭，旧关系永不出场。

### 8.5 典型案例

**坏案例**：一个 `ack_required` 通知没有 deadline，也没有 expired / escalation 路径。

后果：

- 这条通知可能永远挂着；
- 系统无法判断它“还活着”还是“已失效”；
- 长时后 notification_buffer 被历史悬挂项慢慢塞满。

## 9. PA7 → G：信息可达性

### 9.1 公理语义

PA7 回答的是：

> 该到某主体的信息，是否真的能在有效期内到达它？

形式化骨架：

\[
\forall i, \forall a,\; right(a,i) \Rightarrow \exists t \le deadline(i): deliver(i,a,t)
\]

### 9.2 主具体化：G5（感知侧）

PA7 的主承载者是 G5 的感知正确性部分：

- G5-R009：local / external 阈值必须区分；
- G5-R010：生命威胁强制唤醒；
- G5-R011：用户直接输入高权限路径；
- G5-R013 / R014：View 的 observer 域与 field 决定的可见性；
- G5-R015：semantic_class 支持。

### 9.3 次具体化：G2 / G3 / G7

- **G2**：assemble 必须把满足注入条件的系统输入组装进上下文；
- **G3**：event-driven 调度保证到达的信息不会因为永不唤醒而失效；
- **G7**：记忆检索保证跨轮次的关键信息不会因为偶然检索失败而“逻辑上到达，认知上没到达”。

### 9.4 若 PA7 失效，会先坏在哪里

- 系统看起来有世界、有消息、有事件；
- 但相关主体就是感知不到；
- 高危事件与普通噪声没有区别；
- 最后系统沦为“世界在变，agent 没感到”。

### 9.5 典型案例

**坏案例**：hazard / evacuation / emergency 仍然走普通通知门槛。

后果：

- agent 处在专注写作或睡眠状态时可能根本收不到撤离信号；
- 系统形式上保持“统一门槛”，但实际上破坏了 T5 的真实耦合。

## 10. PA8 → G：语义进展

### 10.1 公理语义

PA8 回答的是：

> 系统是不是在“有语义地推进”，还是只是在形式上不断触发？

形式化骨架：

\[
\exists P: S \to \Pi
\]

若存在循环 \(C\)，且

\[
\sum_{s_k \to s_{k+1} \in C} \Delta P = 0
\]

则必须存在无需人工干预的 escape mechanism。

### 10.2 主具体化：G4

PA8 的主具体化就是 **G4 开放演化性**：

- G4-R001：定义 progress projection；
- G4-R002：追踪 `last_progress_at`；
- G4-R003：逃离机制必须存在；
- G4-R004：软逃离；
- G4-R005：强逃离；
- G4-R006：Obligation 循环检测；
- G4-R007：消息流动 ≠ 语义进展。

### 10.3 次具体化：G8-R009

G8-R009 不是 PA8 的抽象承载者，而是：

> G4 理论要求在工程层的一个正式实现。

这点必须和“PA8 的主具体化是 G4”明确区分。

### 10.4 若 PA8 失效，会先坏在哪里

系统最危险的错觉就是：

- tick 还在走；
- notification 还在流；
- agent 还在回复；
- 但 deliverable、agenda、obligation、world effect 等关键投影完全不变。

这就是“假活性”。

### 10.5 典型案例

**坏案例**：两个 agent 在强任务场里反复 `send_message / short_reply / acknowledge`，但没有任何 PROGRESS_ACTIONS。

后果：

- 系统表面极其活跃；
- 任务实质毫无推进；
- 若没有 G4，系统会把这种活跃误判成“工作正常”。

## 11. PA9 → G：观察—行动耦合

### 11.1 公理语义

PA9 回答的是：

> 主体能不能既观察到环境，又通过行动真实改变环境？

形式化骨架：

\[
\forall a \in Agents,\; \exists obs_a, act_a
\]

使得

\[
observable(env \to a) \land net\_effect(a \to env) \neq 0
\]

### 11.2 主具体化：G5

PA9 的主承载者是 **G5 语义有效耦合** 的整体：

- Field / Activity 分开建模；
- `field × activity × agent_state` 共同决定运行时行为；
- RuntimeEnvelope 桥接；
- 动作可达域、co-located 的 place_id；
- obligation / agenda / closeCondition / 强任务 session；
- LLM 不能直接输出可写 Activity JSON；
- 从输入门控到输出能力形成闭环。

### 11.3 次具体化：G0 / G2 / G6

- **G0**：共享世界结构是 PA9 的存在论前提；
- **G2**：G2-R005 / R005a 把“行动对世界产生净效果”落实成端到端闭环；
- **G6**：ActionReceipt 防止 agent 把“想做成了”误当成“真的做成了”。

### 11.4 若 PA9 失效，会先坏在哪里

系统会出现两种退化：

1. 只观察，不行动：agent 成为纯阅读器；
2. 只行动，不反馈：agent 成为盲执行器。

二者都不是真正的“多主体—世界系统”。

### 11.5 典型案例

**坏案例**：agent 输出“我已经更新了 agenda”，但系统没有 `intent → emit → route → R_A^apply → receipt` 闭环。

后果：

- agent 记忆认为世界已变化；
- 世界侧实际上没有写入；
- 于是 T5 和 T2 一起破坏，最后 G6 也坏。

## 13. 结论：这张映射表真正说明了什么

这份补完版 `PA→G` 映射表，最终说明的是三件事：

### 13.1 PA 不是“中间名词层”，而是结构约束层

PA 层不是为了好看而夹在 T 与 G 中间。

它真正的作用是：

- 把 T 层的“目标语言”，翻译成结构上必须满足的约束；
- 再把这些结构约束，下推成当前粒子系统里的具体系统目标。

### 13.2 G 层不是平铺列表，而是按 PA 被分组的落地层

看似很多 G 目标，实则不是横向堆砌，而是：

- G0 / G2 在落 PA1；
- G1 / G6 在落 PA5、PA6；
- G3 在落 PA2、PA3、PA4、PA6；
- G4 在落 PA8；
- G5 在落 PA7、PA9；
- G7 是当前 LLM 系统对 PA2 / PA4 / PA5 的特化补充；
- G8 是对既有 PA 的工程增强，而不是新公理。

### 13.3 这张表本质上是在限制复杂度

它的真正作用不是“增加理论层数”，而是：

> 任何新增机制，都必须回答：它到底在具体化哪条 PA？

若答不出来，就不该占据高层位置。

这正是整套体系防止无限补丁化、无限概念膨胀的核心手段。

---

## 14. 一页版极简结论

如果只用一句话概括：

- **T 层**规定系统最终必须保住什么；
- **PA 层**规定为了保住这些目标，结构上至少要满足什么；
- **G 层**规定这些结构条件在当前异步耦合粒子系统里如何真正落地。

而 `PA→G` 映射表的作用，就是把这条“抽象约束 → 当前系统目标”的落地通路彻底钉死。
