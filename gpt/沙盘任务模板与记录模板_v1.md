# 《沙盘任务模板与记录模板 v1》

> 版本：v1  
> 定位：本文件用于配套《基于目标—公理—要求体系的沙盘推演与完备验证方法 v1》，给出**可直接执行**的任务模板与记录模板。  
> 目标：把“沙盘推演”从讨论性案例，收束为一套**可编号、可追溯、可复验、可归档**的正式验证流程。  
> 适用范围：面向采用“三层目标—公理—要求体系”的任何待测系统；其中模板中的 **T/PA 通用字段**可跨体系复用，**G/实现字段**随具体体系变化而替换。

---

## 0. 使用原则

本模板遵循以下五条原则：

1. **分层原则**：每个任务与每条结论，都必须落到 `T / PA / G / 任务 / 证据` 某一层或多层映射上。
2. **单元原则**：最小判定单元不是“整个系统”，而是“**一个任务—一个验证对象**”。
3. **双重验证原则**：既验证待测系统，也验证目标—公理—要求体系本身是否完备。
4. **证据优先原则**：所有满足/不满足结论，必须以文本依据、状态演化证据与推理链为基础。
5. **保守归因原则**：若失败结论无法可靠归因，则不得强行上收为高层失败；应标记为“暂不可判定”或“需继续下钻”。

## 1. 核心术语与对象

## 1.1 核心对象

- **待测系统（SUT, System Under Test）**：被推演和验证的系统。
- **验证对象（Validation Target）**：被某任务直接验证的目标，可是某条目标、某条公理、某条要求、某个状态机、某个机制、某个对象。
- **任务（Task）**：为触发并验证特定性质而设计的抽象测试单元。
- **场景（Scenario）**：任务在某组初始条件、触发序列与环境约束下的一次实例化。
- **推演记录（Record）**：对一个场景的完整证据化演化记录。
- **判定（Verdict）**：对某任务—验证对象对给出的满足性结论。
- **根因链（Root Cause Chain）**：从失败现象向上追到要求、公理、目标，或向下追到具体机制缺陷的因果链。
- **体系缺口（Framework Gap）**：当现有目标—公理—要求体系无法解释某类失败时，对体系本身作出的缺口判定。

## 1.2 层级对象

- **T 层（目标层）**：回答系统最终必须保住什么。
- **PA 层（抽象公理层）**：回答为了保住 T 层，结构上必须满足什么。
- **G 层（当前体系要求层）**：回答在当前具体体系中，PA 如何被具体化。
- **任务层**：回答用什么测试任务去触发和观察这些性质。
- **证据层**：回答凭什么得出结论。

## 1.3 任务类型

- **一般性任务（GEN）**：验证 T/PA，去掉当前体系字段名后仍然成立。
- **体系绑定任务（SYS）**：验证当前 G 层、当前 schema、当前对象结构、当前时序规则。
- **混合任务（HYB）**：表面上依赖当前体系，但其判定目的落在 T/PA；需同时记录通用部分与体系绑定部分。

## 2. 文档之间的关系

本文件包含两类模板：

1. **沙盘任务模板**：用于设计任务。
2. **沙盘记录模板**：用于记录一次任务实例化后的完整推演过程。

两者关系如下：

```text
验证映射表（回答“测什么”）
    ↓
沙盘任务模板（回答“怎么设计任务”）
    ↓
沙盘记录模板（回答“这次怎么跑、跑出了什么”）
    ↓
汇总矩阵 / 根因表 / 体系缺口表（回答“最终结论是什么”）
```

## 3. 编号与命名规范

## 3.1 任务编号

推荐格式：

```text
[TASK_TYPE]-[LAYER]-[DOMAIN]-[SEQ]
```

其中：

- `TASK_TYPE ∈ {GEN, SYS, HYB}`
- `LAYER ∈ {T, PA, G, TG, PAG, FULL}`
- `DOMAIN ∈ {LEGAL, BOUND, RECOVER, PROGRESS, COUPLE, TRACE, ENGINEER, ...}`
- `SEQ`：三位或四位流水号

示例：

- `GEN-PA-BOUND-003`
- `SYS-G-TRACE-014`
- `HYB-FULL-COUPLE-021`

## 3.2 场景编号

推荐格式：

```text
[任务编号]-SCN-[两位序号]
```

示例：

- `SYS-G-TRACE-014-SCN-01`
- `GEN-PA-PROGRESS-007-SCN-03`

## 3.3 记录编号

推荐格式：

```text
[场景编号]-REC
```

示例：

- `GEN-PA-PROGRESS-007-SCN-03-REC`

## 4. 判定枚举与证据等级

## 4.1 满足性判定枚举

统一使用以下 5 类：

- **P1 完全满足**
- **P2 部分满足**
- **P3 概率性不满足**
- **P4 完全不满足**
- **P5 暂不可判定**

> 注意：  
> - “概率性不满足”不是“偶然失败”，而是说明在某些可重复出现的条件下缺乏结构性保障。  
> - 一旦判定为 P3，原则上必须继续下钻到更底层要求或公理。

## 4.2 证据等级枚举

统一使用以下 4 级：

- **E0 文本级**：只有规范文本与逻辑推演，没有状态演化证据。
- **E1 场景级**：有明确前提、事件序列与状态变化，但仍主要是手工推演。
- **E2 轨迹级**：有逐步演化轨迹、状态快照、分支对照或形式化表格。
- **E3 复验级**：同一任务在多个场景、多个参数或多次复演下，结论稳定一致。

> 正式归档建议：  
> - 对 G 层具体要求的“完全满足/完全不满足”结论，证据至少达到 **E1**；  
> - 对高层目标和公理的最终归档结论，证据建议达到 **E2**；  
> - 若要把某一类失败上升为体系缺口，建议至少达到 **E2**，最好达到 **E3**。

## 7. 任务与记录之间的对应关系

一个任务可以对应：

- **1 个场景**：用于最小验证；
- **多个场景**：用于覆盖边界条件、参数扫描、对照分支；
- **一组场景簇**：用于支撑高层结论或体系缺口判定。

建议对应关系如下：

| 目标 | 最低要求 |
|---|---|
| 验证某条 G 要求是否存在明显失败 | ≥ 1 个标准场景 |
| 验证某条 PA 是否成立 | ≥ 2 个不同结构场景 |
| 把某失败上升为体系缺口 | ≥ 2 个不同任务域 + ≥ 1 个对照分支 |
| 把某结论归档为“完全满足” | ≥ 1 个主场景 + ≥ 1 个边界场景 |

## 8. 常用任务族与建议模板实例

## 8.1 一般性任务族

### A. 合法性任务族
目标：验证 T2 / PA5 / G1 一类问题  
典型问题：
- 双真源
- 非法跨边界写
- 基线被污染
- 值域越界
- 非法 effect 状态

### B. 有界性任务族
目标：验证 T3 / PA2 / G3 一类问题  
典型问题：
- 队列无限增长
- memory 无界
- obligations 长期不终局
- context 只增不减

### C. 恢复性任务族
目标：验证 T3 / PA3 / PA4 一类问题  
典型问题：
- energy 耗尽后无法恢复
- next_wakeup 链断裂
- 被打断后无法重新进入
- 掉线恢复失败

### D. 进展性任务族
目标：验证 T4 / PA8 / G4 一类问题  
典型问题：
- 空转循环
- 形式活动但无语义进展
- recovering / working 微震荡
- 卡在 blocked obligation 但系统仍形式活跃

### E. 耦合性任务族
目标：验证 T5 / PA7 / PA9 / G5 一类问题  
典型问题：
- 有权信息到达不了主体
- 动作没有世界效果
- 低头族悖论
- hazard 强制唤醒缺失

## 8.2 体系绑定任务族

### A. 时序闭环任务
目标：验证 emit → route → apply → receipt 等闭环  
典型问题：
- 动作声称成功但无 receipt
- receipt 迟到时记忆先写成成功
- close 事件后栈未清理

### B. 真源与账本任务
目标：验证 Global Event Log / WritebackLedger / runtime cache 的角色边界  
典型问题：
- Ledger 变第二真源
- replay 后状态不一致
- crash 后无法重建 durable 状态

### C. 前景与活动任务
目标：验证 ForegroundStack / ActivityInstance / baseline_attention  
典型问题：
- 单值前景场遗留
- 栈返回失败
- 栈顶场已关闭但仍被视为前景
- 场/活动污染 baseline

### D. 工程异常任务
目标：验证 G8 一类工程鲁棒性  
典型问题：
- 掉线
- 重试
- 重放
- 延迟
- split-brain
- fanout 过载
- 灾害广播

## 9. 任务与记录的校对清单

## 9.1 任务编制前校对

- [ ] 该任务的验证对象只有一个主焦点，不是混合无边界大杂烩。
- [ ] 该任务能向上追到至少一条 PA 和至少一个 T。
- [ ] 若该任务是 SYS 型，已明确其依赖的当前体系对象与字段。
- [ ] 若该任务是 GEN 型，去掉当前体系字段名后仍成立。
- [ ] 判定规则足够精确，避免“看起来差不多”的主观判断。
- [ ] 失败后的追溯入口非空。

## 9.2 记录归档前校对

- [ ] 场景是否严格遵守任务前提，若偏离是否明确记录。
- [ ] 文本依据是否与当前版本一致，避免混用旧版术语。
- [ ] 初始状态是否已经通过合法性检查。
- [ ] 逐步推演是否每一步都有“读什么—按什么规则—写回哪里”的说明。
- [ ] 判定是否直接对应到条目，而不是只写“感觉失败了”。
- [ ] 若判定为 P3，是否继续追到了更底层问题。
- [ ] 若怀疑体系缺口，是否说明为何原体系无法解释该失败。
- [ ] 结论是否与证据等级匹配。

## 10. 从任务到汇总报告的最小链条

推荐的最小链条如下：

```text
任务模板
  → 场景实例化
    → 逐步推演记录
      → 单场景判定
        → 多场景汇总
          → 失败根因表
            → 体系缺口审查表
              → 最终验证报告
```

若缺任一环节，则沙盘推演只能算“局部探索”，不能算“完备验证”。

## 12. v1 结语

本文件不负责定义“目标—公理—要求体系本身”，而负责把该体系转化为**可以执行、可以记录、可以复核**的任务与记录模板。  
因此，它的核心价值不在于增加更多规则，而在于保证：

- 任意一个任务都知道自己在测什么；
- 任意一次失败都知道应该往哪里追；
- 任意一份记录都能进入统一归档体系；
- 任意一次体系修正都能回看它是被哪些场景逼出来的。
