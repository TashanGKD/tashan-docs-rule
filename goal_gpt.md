

# 我建议的“目标分级树”

我建议不要把所有目标混成一层，而是分成 6 级。
这 6 级不是“章节”，而是**优先级和依赖关系**。

## G0. 裁剪目标

**目标**：系统只保留对“最小无限演化”必要的东西。
这是最高层判准，不是系统内变量。删掉某对象/规则后，如果不会破坏闭环、长期稳定、恢复能力，它就不该占高层。这个标准正是 GPT 分析里提出的“目标层”。

---

## G1. 存在性目标

**目标**：系统首先必须是一个“可定义的耦合粒子系统”。
也就是：有粒子、有状态空间、有时间、有演化函数、有耦合接口。
Sonnet 的 P1–P6 和 GPT 的“公理核”都在服务这个目标。

它回答的是：
**系统能不能存在。**

---

## G2. 闭环目标

**目标**：系统每次交互都必须形成完整闭环，而不是描述性语句。
F×A 模型把这个标准说得最清楚：一条规则必须能回答六问——到达、看见、中断、输出、写回、再调度。少一问就不是闭环规则。

它回答的是：
**系统能不能真正“跑起来”。**

---

## G3. 长时稳定性目标

**目标**：系统在理想条件下可以无限时间演化，而不会在有限时间内崩掉。
Sonnet 的 C1–C4 本质上就是这一层：能量恢复路径、记忆有界、唤醒链不断、耦合缓冲有终局。

它回答的是：
**系统能不能长期跑下去。**

---

## G4. 功能有效性目标

**目标**：系统不仅“活着”，还要能对上下文产生有效响应，形成真实耦合。
这层对应：

* Field × Activity 真正影响行为
* local / external 阈值分流
* action legality
* 物理场 + 前景场动作并集
* 当前活动真正影响能量与唤醒。

它回答的是：
**系统能不能按我们想要的方式工作。**

---

## G5. 可追溯性目标

**目标**：状态变更必须能重建、能审计、能重放。
这层对应：

* Global Event Log 唯一真源
* 可重放
* 审计视图是派生，不是第二本体。
  Sonnet 的 P7 和 GPT 的“可回放公理”都在服务这个目标。

它回答的是：
**系统能不能被解释、检查、恢复。**

---

## G6. 工程鲁棒性目标

**目标**：在真实计算机系统里，系统不因为数值问题、并发、掉线、重连、分布式异常而失控。
这层对应：

* Hysteresis Lock
* WritebackLedger / PendingIntent
* split-brain 防护
* busy-loop 防护
* extreme Δt 截断
* lease / queue / overload 等。

它回答的是：
**系统在现实实现中能不能稳。**

---

# 这 6 级目标之间的关系

它们不是并列的，而是有依赖顺序：

[
G0 \rightarrow G1 \rightarrow G2 \rightarrow G3 \rightarrow G4 \rightarrow G5 \rightarrow G6
]

更口语一点：

* 没有 **G1**，系统根本没定义；
* 没有 **G2**，系统不能完整跑一轮；
* 没有 **G3**，系统跑几轮就死；
* 没有 **G4**，系统虽然活着，但行为无效；
* 没有 **G5**，系统不可解释、不可恢复；
* 没有 **G6**，系统理论上成立，但工程上不稳。

---

# 这样一来，“要求”应该怎么挂？

我建议以后每条要求都带两个标签：

* **主目标**：它主要服务哪个目标
* **次目标**：它还顺带支持哪些目标

举几个最典型的例子：

### 例 1：粒子三元组 `P=(V_P, View_P, R_P)`

* 主目标：**G1 存在性**
* 次目标：G2 闭环
  因为没有三元组，系统都无法定义。

### 例 2：写隔离

* 主目标：**G1 存在性**
* 次目标：G2 闭环、G5 可追溯
  因为没有写隔离，就没有干净的耦合接口和审计边界。

### 例 3：`next_wakeup` 永不为 null

* 主目标：**G3 长时稳定性**
* 次目标：G2 闭环
  因为这直接保证 time-driven 链不断。

### 例 4：energy 有界且有恢复路径

* 主目标：**G3 长时稳定性**
* 次目标：G4 功能有效性
  因为没有恢复路径，agent 最终会停机。

### 例 5：local / external 阈值分流

* 主目标：**G4 功能有效性**
* 次目标：G2 闭环
  因为这决定系统是否能正确区分“当前场内部信号”和“外部打扰”。

### 例 6：Global Event Log 唯一真源

* 主目标：**G5 可追溯性**
* 次目标：G1 存在性、G6 工程鲁棒性
  因为它决定了系统能否重建与审计。

### 例 7：Hysteresis Lock

* 主目标：**G6 工程鲁棒性**
* 次目标：G3 长时稳定性
  因为它不是理想闭环硬前提，而是防微震荡的工程增强。

# 所以你接下来最合适的工作流，应该是：

## 第一步：先定目标树

也就是我上面这 6 级目标。

## 第二步：把所有要求挂到目标树上

每条要求都问：

* 它主要服务哪个目标？
* 没有它，会破坏哪个目标？

## 第三步：再在每个目标下面分“硬要求 / 可选增强”

例如：

### G3 长时稳定性

* 硬要求：

  * energy bounded + recovery
  * next_wakeup 非空
  * memory bounded
  * notification terminal
* 增强要求：

  * Hysteresis Lock
  * wakeup_refractory
  * sleep_debt
    其中前者是“理想闭环”，后者是“工程增强”。

---

# 我建议你最终把“要求总表”升级成这种形式

不要再只是：

* R-001
* R-002
* R-003

而应该变成：

* `R-001`
* `主目标 = G1`
* `次目标 = G2`
* `前置 = 无`
* `最小实现 = ...`
* `是否工程层 = 否`

这样以后你一看某条要求，就知道：

* 它到底在保什么
* 是硬要求还是增强项
* 该放在理想主干还是工程补丁里
